의존성 주입(DI, Dependency Injection)은 객체 지향 프로그래밍에서 객체가 필요로 하는 의존 객체를 직접 생성하지 않고, 외부에서 생성자나 메소드를 통해 전달받는 디자인 패턴입니다.

이 패턴의 핵심 목적은 객체 간의 결합도를 낮추고, 제어의 역전을 실현하는 것입니다.

## 1. 기술적 차이 비교

객체 Client가 객체 Service를 사용하는 상황을 가정해 보겠습니다.

### DI 미적용

클래스 내부에서 직접 의존 객체를 생성하는 방식입니다.

```java
public class Client {
    private Service service = new ServiceImpl(); 

    public void doSomething() {
        service.execute();
    }
}
```

- 문제점: Client는 ServiceImpl 이라는 구체적인 클래스에 강하게 결합되어 있습니다. 만약 ServiceImpl을 다른 구현체로 변경하려면 Client 코드를 직접 수정해야 합니다. 이는 OCP(개방-폐쇄 원칙)를 위배합니다.

### DI 적용

클래스 내부에서 생성하지 않고, 생성자를 통해 외부로부터 주입받습니다. 이때 구체 클래스가 아닌 인터페이스에 의존합니다.

```java
public class Client {
    private Service service;

    public Client(Service service) {
        this.service = service;
    }

    public void doSomething() {
        service.execute();
    }
}

// 외부 설정
Service serviceImpl = new ServiceImpl();
Client client = new Client(serviceImpl);
```

- 특징: Client는 어떤 Service 구현체가 들어올지 알 필요가 없습니다. 오직 인터페이스 명세에만 의존합니다. 이를 통해 DIP(의존 관계 역전 원칙)를 준수하게 됩니다.

## 2. 의존성 주입 방식

1. 생성자 주입
    - 객체 생성 시점에 생성자 인자로 의존성을 주입합니다.
    - 의존하는 객체가 null이 아님을 보장할 수 있고, 불변성을 유지할 수 있어 가장 권장되는 방식입니다.
2. 세터 주입
    - Setter 메소드를 통해 의존성을 주입합니다.
    - 객체 생성 이후에 의존성을 변경해야 하거나, 선택적인 의존성일 경우 사용합니다.
3. 인터페이스 주입
    - 의존성 주입을 위한 별도의 인터페이스를 정의하고 구현하여 주입합니다.

## 3. DI를 사용하는 기술적 이유

1. 결합도 감소
    - 객체 간의 의존 관계가 느슨해져, 한 클래스를 수정할 때 다른 클래스에 미치는 영향이 최소화됩니다.
2. 단위 테스트 용이성
    - 테스트 대상 객체가 의존하는 객체를 실제 객체 대신 모의 객체로 쉽게 대체하여 주입할 수 있습니다. 이를 통해 고립된 테스트 환경을 구축할 수 있습니다.
3. 가독성 및 재사용성 향상
    - 객체의 생성 책임과 사용 책임이 분리되어, 코드가 명확해지고 유지보수가 용이해집니다.

## 정리

의존성 주입이란 객체 내부에서 new 연산자로 의존 객체를 생성하는 것을 피하고, 외부에서 생성된 객체를 인터페이스 타입으로 주입받아 사용하는 아키텍처 패턴입니다.
