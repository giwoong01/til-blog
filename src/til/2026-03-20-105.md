자바 프로그램의 실행 흐름은 “한 번 작성하면, 어디서든 실행된다”는 자바의 철학을 가능하게 하는 핵심 과정입니다.

이 과정의 중심에는 코드를 운영체제에 맞게 번역해 주는 JVM(Java Virtual Machine)이 있습니다.

크게 컴파일 단계와 실행 단계로 나눌 수 있으며, 구체적인 흐름은 다음과 같습니다.

## 1. 소스 코드 작성

개발자가 자바 언어의 문법에 맞게 소스 코드를 작성합니다. 이 파일은 .java 확장자를 가집니다. (예: Main.java)

## 2. 컴파일

컴퓨터(OS와 CPU)는 사람이 쓴 자바 소스 코드를 바로 이해할 수 없습니다. 따라서 자바 컴파일러(javac)가 .java 파일을 읽어서 바이트코드로 변환합니다.

- 결과물: .class 확장자를 가진 파일이 생성됩니다. (예: Main.class)
- 특징: 이 바이트코드는 0과 1로 이루어진 기계어가 아닙니다. 특정 운영체제에 종속되지 않는 ‘중간 언어’이며,오직 JVM만 읽고 해독할 수 있습니다.

## 3. 클래스 로드

컴파일된 프로그램을 실행(예: java Main 명령어를 입력)하면, OS가 JVM에게 실행에 필요한 메모리를 할당해 줍니다.

- 이때 JVM 내부의 클래스 로더가 동작하여, 필요한 .class 파일들을 하드 디스크에서 찾아 읽어 들입니다.
- 읽어 들인 클래스의 메타데이터(클래스 이름, 메소드 정보, static 변수 등)를 검증한 후, JVM의 메모리 공간인 런타임 데이터 영역에 적재합니다.

## 4. 실행

메모리에 적재된 바이트코드를 실제로 실행하는 단계입니다. JVM의 실행 엔진은 바이트코드를 컴퓨터가 즉시 실행할 수 있는 네이티브 기계어로 실시간 번역하며 명령을 수행합니다.

현대의 JVM은 성능 최적화를 위해 다음 두 가지 방식을 혼합하여 사용합니다.

- 인터프리터: 바이트코드를 한 줄씩 순차적으로 읽어서 번역하고 실행합니다. (시작은 빠르지만 전체적인 실행 속도는 느린편입니다.)
- JIT 컴파일러: 인터프리터의 단점을 극복하기 위한 기술입니다. 실행 엔진이 코드를 지켜보다가 “자주 반복해서 실행되는 코드”를 발견하면, 해당 바이트코드를 통째로 기계어로 컴파일해 버리고 캐시에 저장합니다. 이후에는 매번 번역하지 않고 캐싱된 기계어를 바로 꺼내써서 실행 속도를 비약적으로 끌어올립니다.

## 5. 메모리 관리

프로그램이 실행되면서 로직에 따라 메모리(Heap 영역)에는 수많은 객체가 생성됩니다. JVM의 가비지 컬렉터는 백그라운드에서 돌면서 더 이상 사용되지 않는 객체들을 찾아내어 자동으로 메모리를 청소해 줍니다. 덕분에 자바 개발자는 C언어처럼 메모리 해제 코드를 직접 작성하지 않아도 됩니다.

## 6. 런타임 데이터 영역의 5가지 구조

자바 프로그램이 실행될 때 운영체제로부터 할당받는 메모리 공간인 런타임 데이터 영역은 철저하게 분업화된 5개의 구역으로 나뉩니다.

이 공간은 크게 ‘모든 쓰레드가 공유하는 영역(2개)’과 ‘각 쓰레드마다 개별적으로 생성되는 영역(3개)’으로 구분합니다.

### 1. 모든 쓰레드가 공유하는 영역

JVM이 시작될 때 생성되며, 프로그램이 종료될 때까지 유지됩니다. 여러 쓰레드가 동시에 접근할 수 있어 동시성 문제에 주의해야 하는 곳입니다.

1. 메소드 영역
    
    “설계도와 공용 데이터가 모여있는 보관소”
    
    - 저장 내용: 클래스 로더가 읽어 들인 클래스의 메타데이터(이름, 접근 제어자 등), 정적 변수, 메소드의 바이트코드, 그리고 런타임 상수 풀이 저장됩니다.
    - 특징: 프로그램 전반에서 공유되어야 하는 데이터들이 올라가므로, 애플리케이션의 뼈대가 되는 정보들이 위치하는 곳입니다.

1. 힙 영역
    
    “실제 데이터(객체)가 생성되고 소멸하는 작업장”
    
    - 저장 내용: 코드에서 new 키워드를 통해 생성된 모든 인스턴스(객체)와 배열이 저장됩니다.
    - 특징: 개발자가 가장 신경 써야 하는 메모리 영역입니다. 이곳에 생성된 객체들은 스택 영역의 변수나 다른 객체의 필드에서 참조를 통해 접근합니다.
    - 관리: 아무도 참조하지 않는 미아 상태의 객체들은 가비지 컬렉터가 주기적으로 힙 영역을 청소하여 메모리를 확보합니다.

### 2. 쓰레드마다 개별 생성되는 영역

새로운 쓰레드가 하나 생성될 때마다 아래의 3가지 영역이 세트로 함께 생성되고, 쓰레드가 종료되면 메모리에서 함께 사라집니다.

1. 스택 영역
    
    “메소드 실행의 흐름을 기억하는 메모장”
    
    - 저장 내용: 메소드가 호출될 때마다 스택 프레임이라는 묶음이 하나씩 생성되어 쌓입니다. 이 안에는 지역 변수, 매개변수, 리턴 값, 연산 중 발생하는 임시 데이터가 저장됩니다.
    - 특징: 메소드 실행이 끝나면 해당 프레임은 스택에서 즉시 삭제됩니다. 재귀 함수를 잘못 짜서 무한루프를 돌면 이 공간이 꽉 차서 StackOverflowError가 발생합니다.

1. PC 레지스터
    
    “현재 쓰레드가 어디까지 실행했는지 가리키는 책갈피”
    
    - 저장 내용: 현재 쓰레드가 실행 중인 JVM 명령어의 메모리 주소를 저장합니다.
    - 특징: 자바는 멀티 쓰레드 환경이므로, CPU가 다른 쓰레드의 작업을 처리하다가 다시 돌아왔을 때 “내가 아까 어디까지 코드를 읽었더라?”를 알기 위해 반드시 필요한 영역입니다.

1. 네티이브 메소드 스택
    
    “자바 외의 언어로 작성된 코드를 위한 특별 구역”
    
    - 저장 내용: 성능 향상이나 하드웨어 제어를 위해 자바가 아닌 C, C++ 등의 기계어로 작성된 함수를 실행할 때 사용하는 공간입니다.
    - 특징: 자바 코드의 실행 흐름과 석이지 않도록 별도의 스택 공간을 두어 관리합니다.

## 정리

실행 흐름

1. 개발자가 .java 작성
2. 컴파일러가 .class (바이트코드)로 변환
3. 클래스 로더가 JVM 메모리에 적재
4. 실행 엔진(인터프리터 + JIT)이 기계어로 실시간 번역하여 실행
5. 다 쓴 메모리는 가비지 컬렉터가 알아서 정리

영역 5가지

- 메소드 영역: 클래스 정보, static 변수 (공유)
- 힙 영역: new로 만든 객체와 배열 (공유, GC의 타겟)
- 스택 영역: 지역 변수, 메소드 실행 흐름 (쓰레드별)
- PC 레지스터: 현재 실행 중인 코드 주소 (쓰레드별)
- 네이티브 스택: C/C++ 등 타 언어 코드 실행 (쓰레드별)
