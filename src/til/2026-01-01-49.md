시스템 간 비동기 연동은 요청을 보내는 시스템이 결과를 즉시 기다리지 않고(Non-blocing), 자신의 작업을 계속 진행할 수 있도록 하는 통신 방식입니다. 시스템 간의 결합도를 낮추고 탄력성을 높이는 데 핵심적인 역할을 합니다.

주요 방식은 크게 메시지 브로커 기반, HTTP 기반, 데이터베이스 기반으로 나눌 수 있습니다.

## 1. 메시지 브로커 기반

중간에 메시지를 보관하고 전달해 주는 중개자(Broker)를 두는 가장 대표적인 비동기 방식입니다.

### A. 메시지 큐

- 개념: 생산자(Producer)가 메시지를 큐에 넣으면, 소비가(Consumer)가 큐에서 메시지를 하나씩 꺼내 처리합니다.
- 특징
    - P2P (Point-to-Point): 하나의 메시지는 오직 하나의 소비자에게만 전달됩니다.
    - 보장성: 소비자가 메시지를 잘 처리했는지 확인할 수 있어 데이터 유실 가능성이 낮습니다.
- 기술: RabbitMQ, ActiveMQ, AWS SQS
- 용도: 이메일 발송, 이미지 리사이징 등 무거운 백그라운드 작업 처리

### B. 발행/구독

- 개념: 생산자가 특정 토픽에 메시지를 발행하면, 해당 토픽을 구독하고 있는 모든 소비자에게 메시지가 전달됩니다.
- 특징
    - 브로드캐스팅: 하나의 이벤트로 여러 시스템(로그 저장, 알림 발송, 분석 등)을 동시에 트리거할 수 있습니다.
    - 느슨한 결합: 생산자는 누가 구독하는지 전혀 몰라도 됩니다.
- 기술: Apache Kafka, Redis Pub/Sub, AWS SNS
- 용도: 이벤트 기반 아키텍처 구현, 실시간 데이터 스트리밍

## 2. HTTP 기반

별도의 미들웨어 없이 HTTP 프로토콜을 활용하여 비동기 처리를 구현하는 방식입니다.

### A. 웹훅 - Callback

- 개념: “작업 다 되면 내가 너한테 전화 줄게” 방식입니다. 클라이언트가 요청을 보내면 서버는 “접수됨”만 응답하고 연결을 끊습니다. 작업이 완료되면 서버가 클라이언트가 미리 등록해 둔 URL로 결과를 전송합니다.
- 특징: 서버 → 클라이언트 방향으로 역방향 호출이 일어납니다.
- 용도: 결제 모듈(PG사 결제 완료 통보), Git Push 알림(CI/CD 트리거)

### B. 폴링

- 개념: 클라이언트가 서버에 작업을 요청하고, 주기적으로 “작업 다 됐니?”라고 상태 확인 API를 호출합니다.
- 특징: 구현이 쉽지만, 불필요한 요청이 많아져 서버와 네트워크 리소스를 낭비할 수 있습니다.
- 용도: 웹훅을 사용할 수 없는 환경이거나, 긴 작업의 진행률을 UI에 보여줘야 할 때

## 3. 데이터베이스 기반

### A. 공유 테이블

- 개념: JOB_QUEUE 같은 테이블을 만들고, A 시스템이 INSERT 하면 B 시스템이 이를 SELECT 해서 처리하고 상태를 UPDATE 합니다.
- 특징: 별도 인프라가 필요 없지만, DB 부하가 심하고 실시간성이 떨어집니다.

### B. CDC

- 개념: DB의 트랜잭션 로그를 감지하여, 데이터 변경이 일어날 때마다 이를 이벤트로 발행합니다.
- 기술: Debezium, Kafka Connect
- 용도: DB 간 데이터 동기화, 캐시 갱신 등 데이터 변경 사항을 다른 시스템에 전파할 때
