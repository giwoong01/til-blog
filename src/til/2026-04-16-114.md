널 오브젝트 패턴은 객체가 존재하지 않아야 할 상황에서 null을 반환하는 대신, “아무 일도 하지 않는” 기본 동작을 정의한 가짜 객체를 만들어 반환하는 소프트웨어 디자인 패턴입니다.

## 1. 해결하려는 문제

보통 함수에서 조건에 맞는 객체를 찾지 못하면 null을 반환합니다. 이 때문에 우리는 개발할 때 수많은 방어 코드를 작성해야 합니다.

```java
// Null Object Pattern 미사용 시
Logger logger = LoggerFactory.getLogger(type);

// 매번 null인지 확인해야 함. 빼먹으면 NullPointerException 발생!
if (logger != null) {
    logger.log("시스템이 시작되었습니다.");
}
```

이러한 if (obj ≠ null) 조건문은 코드를 지저분하게 만들고, 개발자가 실수로 체크를 누락하면 NullPointerException을 발생시킵니다.

## 2. 동작 원리

널 오브젝트 패턴은 다형성을 활용하여 이 문제를 해결합니다.

1. 인터페이스 정의: 실제 동작을 할 객체와 가짜 객체가 공통으로 구현할 인터페이스를 만듭니다.
2. 실제 객체 구현: 실제 비즈니스 로직을 수행하는 클래스를 만듭니다.
3. 널 객체 구현: 인터페이스의 메소드를 구현하되, 내부를 텅 비워두거나 안전한 기본값만 반환하도록 만듭니다.

```java
// 1. 공통 인터페이스
public interface Logger {
    void log(String message);
}

// 2. 실제 동작하는 객체
public class ConsoleLogger implements Logger {
    @Override
    public void log(String message) {
        System.out.println("로그 출력: " + message);
    }
}

// 3. 아무것도 하지 않는 널 객체
public class NullLogger implements Logger {
    @Override
    public void log(String message) {
        // 아무 일도 하지 않음 (Do nothing)
    }
}
```

이제 이 패턴을 적용하면 클래스언트 측의 코드가 깔끔해집니다.

```java
// Null Object Pattern 사용 시
Logger logger = LoggerFactory.getLogger(type); // 없으면 NullLogger 반환

// null 체크를 할 필요가 전혀 없음!
// logger가 ConsoleLogger면 출력되고, NullLogger면 조용히 지나감.
logger.log("시스템이 시작되었습니다.");
```

## 3. 장점과 단점

### 장점

- 코드 가독성 향상: 불필요한 null 체크 로직이 사라져 핵심 비즈니스 로직에만 집중할 수 있습니다.
- 안정성: 개발자의 실수로 인한 NullPointerException 발생 위험이 근본적으로 차단됩니다.
- 객체 지향적 설계: 클라이언트는 반환된 객체가 진짜인지 가짜인지 신경 쓸 필요 없이 인터페이스만 보고 일관되게 메시지를 보낼 수 있습니다.

### 단점

- 클래스 폭발: 단순히 null을 피하기 위해 껍데기뿐인 Null 클래스를 따로 만들어야 하므로 관리해야 할 클래스 개수가 늘어납니다.
- 오류 은닉: 진짜로 문제가 생겨서 객체를 찾지 못해 예외가 발생해야 하는 상황인데, 널 객체가 아무 일도 없었던 것처럼 묵묵히 넘어가 버려서 나중에 버그의 원인을 찾기 힘들어질 수 있습니다.
