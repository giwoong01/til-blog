자바 제네릭을 공부할 때 가장 헷갈리는 개념 중 하나가 바로 변성입니다.

타입 A가 타입 B의 하위 타입일 때, 제네릭 타입 List<A>와 List<B> 사이에 어떤 상속 관계가 성립하는가를 정의하는 규칙입니다. 자바에서는 와일드카드(?)를 사용하여 이를 제어합니다.

## 1. 무공변 (Invariance)

“너와 나는 아무런 남남이다.”

- 개념: 타입 A가 타입 B의 하위 타입이더라도, List<A>와 List<B>는 아무런 상속 관계가 없습니다.
- 자바의 기본 상태: 자바의 모든 제네릭은 기본적으로 무공변입니다.
- 이유 (타입 안전성): 만약 이게 허용된다면, 런타임에 심각한 타입 에러(ClassCastException)가 발생할 수 있기 때문에 컴파일러가 원천 차단합니다.

```java
// String은 Object의 하위 타입이지만,
// List<String>은 List<Object>의 하위 타입이 아닙니다.
List<Object> objects = new ArrayList<String>(); // ❌ 컴파일 에러 발생!
```

## 2. 공변 (Convariance)

“네가 내 상위 타입이면, 나도 네 제네릭의 하위 타입이다.” (방향 일치)

- 개념: 타입 A가 타입 B의 하위 타입일 때, List<A>도 List<B>의 하위 타입으로 인정해 주는 것입니다.
- 문법: ? extends T (상한 경계 와일드카드)
- 특징 (Read-Only): 데이터를 읽을 수만 있고, 쓸 수는 없습니다. (단, null은 제외)

```java
// Integer는 Number의 하위 타입이므로 성립합니다.
List<? extends Number> list = new ArrayList<Integer>(); // ✅ 정상 작동

Number n = list.get(0); // ✅ 읽기 가능 (항상 Number 거나 그 하위 타입임을 보장)
list.add(10);           // ❌ 쓰기 불가능 (실제 리스트가 Double용일 수도 있기 때문)
```

## 3. 반공변 (Contravariance)

“네가 내 상위 타입이면, 오히려 내가 네 제네릭의 상위 타입이다.” (방향 역전)

- 개념: 타입 A가 타입 B의 하위 타입일 때, 상속 관계가 뒤집혀서 List<B>가 List<A>의 하위 타입처럼 취급되는 것입니다.
- 문법: ? super T (하한 경계 와일드카드)
- 특징 (Write-Only): 데이터를 안전하게 쓸 수 있지만, 읽을 때는 타입이 보장되지 않습니다. (꺼낼 때는 무조건 Object 타입으로만 나옵니다.)

```java
// Number는 Integer의 상위 타입이므로 성립합니다.
List<? super Integer> list = new ArrayList<Number>(); // ✅ 정상 작동

list.add(10);             // ✅ 쓰기 가능 (Integer 또는 그 하위 타입은 무조건 들어감 보장)
Integer i = list.get(0);  // ❌ 읽기 불가능 (꺼낸 값이 Number일지 Object일지 알 수 없음)
Object o = list.get(0);   // ✅ Object로만 받을 수 있음
```

## 정리

| 구분   | 키워드        | 관계           | 가능 작업           | 주요 용도                |
| ------ | ------------- | -------------- | ------------------- | ------------------------ |
| 무공변 | <T>           | 관계 없음      | 읽기/쓰기 모두 가능 | 일반적인 제네릭 사용     |
| 공변   | <? extends T> | 상속 방향 일치 | 읽기 전용 <Read>    | 데이터 제공자 (Producer) |
| 반공변 | <? super T>   | 상속 방향 역전 | 쓰기 전용 <Write>   | 데이터 소비자 (Consumer) |

이러한 공변/반공변을 실무에서 언제 써야 할지 헷갈릴 때, Effective Java의 저자 조슈아 블로크가 제안한 유명한 공식이 있습니다. 바로 PECS (Producer-Extends, Consumer-Super)입니다.

- 생산자(Producer): 외부로 데이터를 꺼내주는(반환하는) 역할이면 extends를 씁니다.
- 소비자(Consumer): 외부에서 데이터를 집어넣는(소비하는) 역할이면 super를 씁니다.
