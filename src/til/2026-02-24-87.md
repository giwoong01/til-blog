경쟁 상태(Race Condition)란 두 개 이상의 쓰레드가 공유 자원에 동시에 접근하여, 그 실행 순서에 따라 결과가 뒤바뀌거나 잘못된 값이 저장되는 현상을 말합니다.

이를 해결하기 위해서는 동시성 제어를 통해 3가지 핵심 요소가 반드시 보장되어야 합니다.

## 1. 원자성

가장 중요한 조건입니다. 공유 자원에 대한 일련의 작업이 “전부 실행되거나, 아예 실행되지 않아야 한다.”는 성질입니다.

- 문제 상황: count++ (증가 연산)
  - 이 코드는 사실 컴퓨터 내부에서 3단계로 쪼개집니다.
    1. Read: 메모리에서 count 값을 읽어온다.
    2. Modify: CPU에서 값을 1 더한다.
    3. Write: 더한 값을 다시 메모리에 쓴다.
  - 쓰레드 A가 2번(더하기)을 하고 3번(쓰기)을 하기 전에, 쓰레드 B가 끼어들어서 옛날 값을 읽어가면 데이터가 유실됩니다.
- 해결: 이 3단계(읽기-수정-쓰기)가 중간에 끊기지 않고 마치 하나의 명령어처럼(Atomic) 한 번에 실행되도록 보장해야 합니다.
- 구현 방법: synchronized 블록, Lock, AtomicInteger 등

## 2. 가시성

한 쓰레드가 공유 자원을 수정했을 때, 다른 쓰레드가 그 변경된 값을 “즉시” 볼 수 있어야 한다는 성질입니다.

- 문제 상황: CPU 캐시
  - 성능을 위해 쓰레드는 메인 메모리가 아니라 각자의 CPU 캐시에서 변수를 가져와 작업합니다.
  - 쓰레드 A가 캐시에서 값을 10으로 바꿨는데, 아직 메인 메모리에 반영하지 않았다면? 쓰레드 B는 여전히 메인 메모리의 0을 보고 작업하게 됩니다.
- 해결: 변경된 값을 캐시에만 두지 말고 즉시 메인 메모리에 반영하거나, 읽어올 때 반드시 메인 메모리에서 읽도록 강제해야 합니다.
- 구현 방법: volatile 키워드, synchronized (락을 해제할 때 메인 메모리에 동기화됨).

## 3. 순서성

컴파일러나 CPU는 성능 최적화를 위해 코드의 실행 순서를 임의로 바꿀 수 있습니다.

- 문제 상황
  - 개발자는 A = 1; B = 2; 순서대로 작성했지만, CPU가 보기에 서로 연관이 없으면 B = 2; A = 1;로 순서를 바꿀 수 있습니다.
  - 단일 쓰레드에서는 문제 없지만, 멀티 쓰레드 환경에서는 이 순서 변경이 치명적인 오류를 일으킬 수 있습니다.
- 해결: 명령어의 실행 순서를 강제하여 예측 가능하게 만들어야 합니다.
- 구현 방법: volatile, synchronized (Happens-Before 관계 성립)

## 4. 구현하기 위한 기술 (Java)

1. Synchronized / Lock
   - 임계 구현을 설정하여 한 번에 하나의 쓰레드만 접근하게 합니다.
   - 원자성 + 가시성 + 순서성을 모두 보장하는 가장 확실한 방법입니다.
2. Volatile
   - 변수를 CPU 캐시가 아닌 메인 메모리에 직접 읽고 쓰게 합니다.
   - 가시성과 순서성은 보장하지만, 원자성은 보장하지 않습니다. (단순 flag 변수에 적합)
3. Atomic 클래스
   - CAS 알고리즘을 사용하여 락 없이 원자성을 보장합니다.

## 정리

경쟁 상태를 막으려면 “누가 작업 중일 땐 건드리지 못하게 하고(원자성), 고치면 바로 다 같이 알아야 하며(가시성), 순서 섞지 말고 차례대로(순서성)” 처리해야 합니다.
