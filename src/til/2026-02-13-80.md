데이터베이스 정규화는 관계형 데이터베이스 설계에서 중복된 데이터를 최소화하고, 데이터의 무결성을 유지하기 위해 테이블을 구조적으로 분해하는 과정을 말합니다.

쉽게 말해, “하나의 테이블에는 그 주제에 맞는 데이터만 남기고, 다른 주제의 데이터는 별도의 테이블로 떼어내는 것”입니다.

## 1. 정규화를 하는 이유: 이상 현상 방지

테이블을 제대로 쪼개지 않고 한 통에 다 몰아넣으면(비정규화 상태), 다음과 같은 이상 현상이 발생합니다.

1. 삽입 이상: 데이터를 넣고 싶은데, 불필요한 데이터까지 억지로 함께 넣어야만 저장이 가능한 상황.
    - 예: 신규 회원을 등록하려는데, 아직 수강 신청한 과목이 없어서 등록이 안 됨
2. 갱신 이상: 중복된 데이터 중 일부만 수정되어 데이터 간의 불일치가 발생하는 상황.
    - 예: 홍길동의 주소를 바꿨는데, 10개 행 중 3개만 바뀌고 7개는 옛날 주소로 남음
3. 삭제 이상: 어떤 정보를 지우려는데, 지우면 안 되는 중요한 정보까지 같이 날아가는 상황
    - 예: 수강 취소를 했더니 학생의 인적 사항까지 삭제됨

## 2. 정규화 단계 (1NF, 2NF, 3NF)

정규화는 여러 단계가 있지만, 실무에서는 보통 제3 정규형(3NF)까지만 진행하는 경우가 많습니다.

### 1. 제1 정규형 (1NF) - 원자성

- 규칙: 테이블의 모든 컬럼은 하나의 값만 가져야 합니다.
- 예시: 취미 컬럼에 “축구, 농구, 야구”라고 쉼표로 구분해서 여러 개를 넣으면 안 됩니다. 이를 별도의 행이나 테이블로 분리해야 합니다.

### 2. 제2 정규형 (2NF) - 부분 함수 종속 제거

- 조건: 1NF를 만족하고, 기본키가 복합키일 때 적용됩니다.
- 규칙: 기본키의 일부분에만 의존하는 컬럼을 별도의 테이블로 분리해야 합니다.
- 예시: 기본키가 (학생번호, 과목코드)인 성적 테이블에 교수명이 들어있다고 칩시다. 교수명은 과목코드만 알면 결정됩니다. 학생번호와 상관이 없습니다. 이럴 때 (과목코드, 교수명)을 따로 떼어내야 합니다.

### 3. 제3 정규형 (3NF) - 이행 함수 종속 제거

- 조건: 2NF를 만족해야 합니다.
- 규칙: 기본키가 아닌 일반 컬럼끼리 의존하는 관계(A → B → C)를 끊어야 합니다.
- 예시: (학번) → (소속학과) → (학과사무실 위치) 관계가 있을 때, 학번을 알면 학과사무실 위치를 알 수 있지만, 사실 이는 소속학과에 종속된 정보입니다. 학과와 위치 정보를 별도 테이블로 분리해야 합니다.

## 3. 장단점

정규화는 무조건 많이 한다고 좋은 것이 아닙니다.

### 장점

- 데이터 중복 최소화: 저장 공간을 절약할 수 있습니다.
- 데이터 무결성 유지: 이상 현상(삽입/삭제/갱신)을 방지하여 데이터가 정확하게 유지됩니다.
- 구조 확장성: 테이블이 명확하게 나뉘어 있어 구조 변경 시 영향 범위가 적습니다.

### 단점

- 조인 연산 증가: 데이터가 여러 테이블에 흩어져 있으므로, 필요한 데이터를 조회할 때 JOIN을 많이 써야 합니다.
- 응답 속도 저하: 조인이 많아지면 쿼리가 복잡해지고 CPU 사용량이 늘어나 읽기 성능이 떨어질 수 있습니다.

## 정리

- 정규화: 이상한 일이 안 생기게 테이블을 쪼개는 것
- 핵심: 중복 제거와 무결성 보장
- 주의점: 너무 쪼개면 조인 때문에 조회가 느려질 수 있습니다. 이럴 땐 반대로 합치는 반정규화를 고려해야 합니다.
