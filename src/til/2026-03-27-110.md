자바의 가비지 컬렉터(GC)는 애플리케이션의 멈춤 현상, 즉 ‘Stop-the-Wrold(STW)’ 시간을 최소화하기 위해 끊임없이 발전해 왔습니다.

GC의 기본 원리인 ‘Mark and Sweep(사용 중인 객체 식별 후, 쓰레기 객체 청소)’을 어떤 방식으로 수행하는냐에 따라 크게 5가지 알고리즘으로 나눌 수 있습니다.

## 1. 시리얼 GC (Serial GC)

- 동작 방식: CPU 코어가 1개이던 시절을 위해 만들어진 가장 단순한 방식입니다. 청소를 할 때 단 1개의 쓰레드만 사용합니다.
- 특징: 청소하는 동안 애플리케이션의 다른 모든 작업이 멈추기 때문에 멈춤 시간이 매우 깁니다.
- 사용처: 시스템 자원이 극도로 제한된 환경이나 데스크톱의 아주 가벼운 애플리케이션에서만 제한적으로 사용됩니다.

## 2. 패러렐 GC (Parallel GC)

- 동작 방식: Serial GC와 기본 원리는 같지만, 청소를 할 때 여러 개의 쓰레드를 동원하여 속도를 높인 방식입니다.
- 특징: 다수의 코어를 활용해 STW 시간을 크게 줄였습니다. 애플리케이션이 멈추는 시간 자체를 줄여 전체적인 처리량을 높이는 데 초점이 맞춰져 있습니다.
- 사용처: Java 8의 기본 GC이며, 백그라운드에서 대량의 데이터를 처리하는 배치 프로그램에 적합합니다.

## 3. CMS (Concurrent Mark Sweep) GC

- 동작 방식: 애플리케이션 쓰레드와 GC 쓰레드가 동시에 실행되도록 하여 STW 시간을 극단적으로 줄이려 시도한 방식입니다.
- 특징: 응답 속도가 중요한 서비스에 널리 쓰였으나, 치명적인 단점이 있었습니다. 청소 후 빈 공간을 모아주는 작업을 기본적으로 하지 않아 메모리 파편화가 심하게 발생했고, CPU 리소스를 너무 많이 소모했습니다.
- 현재 상태: 결국 단점을 극복하지 못하고 Java 9부터 퇴역 처리 되었으며, Java 14에서 완전히 삭제되었습니다.

## 4. G1 (Garbage First) GC

- 동작 방식: 기존처럼 메모리를 물리적인 Young/Old 영역으로 넓게 나누지 않고, 전체 힙 메모리를 바둑판 모양의 일정한 크기를 가진 지역들로 잘게 쪼갭니다.
- 특징: 이름 그대로 가비지가 가장 꽉 찬 지역부터 먼저 집중적으로 청소합니다. 개발자가 “STW 시간을 200ms 이하로 맞춰줘”라고 설정하면, G1 GC가 그 시간에 맞춰 청소할 Region의 개수를 동적으로 조절합니다.
- 사용처: Java 9 이후의 기본 GC이며, 대용량 메모리와 멀티 코어 환경에서 가장 널리 사용되는 현재의 표준입니다.

## 5. ZGC (Z Garbage Collector)

- 동작 방식: G1 GC에서 한 단계 더 진화하여, 메모리 위치를 나타내는 포인터 자체에 객체의 상태 정보를 숨겨두는 Colored Pointers 등 최신 기술을 사용합니다.
- 특징: 수백 메가바이트부터 수 테라바이트 단위의 거대한 힙 메모리를 사용하더라도, STW 시간을 1ms이하(Java 21 기준)로 유지하는 엄청난 성능을 자랑합니다. 청소와 압축 과정 대부분이 애플리케이션 쓰레드와 동시에 돌아갑니다.
- 사용처: 극단적으로 짧은 응답 시간이 요구되는 최신 대규모 금융, 게임, 트레이딩 시스템 등에서 점점 도입이 늘고 있습니다. (Java 15에서 정식 기능으로 승격되었습니다.)
