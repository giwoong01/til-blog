DNS(Domain Name System)는 사람이 읽을 수 있는 도메인 이름을 컴퓨터가 통신하는 데 사용하는 IP 주소로 변환해 주는 시스템입니다.

인터넷의 전화번호부 역할을 하는 분산 데이터베이스 시스템으로, 전 세계에 흩어져 있는 수많은 서버가 계층적으로 연결되어 작동합니다.

## 1. DNS의 계층 구조

DNS는 단일 서버가 모든 정보를 가지고 있는 것이 아니라, 역트리 구조로 관리 권한이 분산되어 있습니다.

1. Root DNS Server (루트 도메인 서버)
   - DNS 계층의 최상위입니다. 전 세계에 13개의 원본 서버가 있으며, .com, .net, .kr 같은 TLD 서버의 주소를 알고 있습니다.
2. TLD Server (Top-Level Domain, 최상위 도메인 서버)
   - 도메인의 가장 끝부분(.com, .org, .kr)을 관리하는 서버입니다. 해당 확장자를 가진 도메인의 Authoritative Nameserver 위치를 알려줍니다.
3. Authoritative Nameserver (권한 있는 네임 서버)
   - 실제 특정 도메인에 대한 구체적인 IP 주소 정보를 저장하고 있는 최종 서버입니다. 도메인 구매 시 설정하는 네임서버가 바로 이것입니다.

## 2. DNS 동작 과정

사용자가 브라우저에 www.example.com을 입력했을 때, IP 주소를 찾아오는 과정은 다음과 같습니다.

1. Local Cache 확인
   - 브라우저나 OS가 이전에 방문했던 기록이나 hosts 파일을 먼저 확인합니다. 없으면 다음 단계로 넘어갑니다.
2. Recursive DNS Server (재귀적 질의 서버) 요청
   - ISP(통신사)나 설정된 Public DNS(구글 8.8.8.8 등)에게 “이 도메인 IP 좀 찾아줘”라고 요청합니다. 이 서버가 클라이언트를 대신해 발로 뛰며 정보를 찾아옵니다.
3. Root Server 질의
   - Recursive 서버가 Root 서버에게 묻습니다. “.com 어디서 관리해?” → Root 서버는 .com TLD 서버의 IP를 알려줍니다.
4. TLD Server 질의
   - Recursive 서버가 .com TLD 서버에게 묻습니다. “[example.com](http://example.com) 누구네 서버야?” → TLD 서버는 example.com을 관리하는 Authoritative 서버의 IP을 알려줍니다.
5. Authoritative Server 질의
   - Recursive 서버가 마지막으로 해당 서버에게 묻습니다. “www.example.com의 진짜 IP가 뭐야?” → 비로소 실제 IP 주소를 반환합니다.
6. 결과 반환 및 캐싱
   - Recursive 서버는 얻어온 IP를 클라이언트에게 전달하고, 다음 요청을 위해 자신의 캐시에 저장합니다.

## 3. 주요 DNS 레코드 타입

DNS 서버는 단순히 IP만 저장하는 것이 아니라, 목적에 따라 다양한 레코드를 저장합니다.

- A(Address Mapping): 도메인을 IPv4 주소로 매핑합니다.
- AAAA (IP Version 6 Address): 도메인을 IPv6 주소로 매핑합니다.
- CNAME (Canonical Name): 도메인의 별칭을 지정합니다. (예: m.naver.com → naver.com)
- MX (Mail Exchanger): 메일 서버의 주소를 지정합니다. 이메일을 보낼 때 사용됩니다.
- NS (Name Server): 해당 도메인을 관리하는 네임서버의 주소를 지정합니다.
- TXT (Text): 텍스트 정보를 저장하며, 주로 도메인 소유권 확인이나 스팸 메일 방지 등에 사용됩니다.

## 정리

- DNS: 도메인 ↔ IP 변환 시스템
- 구조: Root → TLD → Authoritative로 이어지는 계층형 분산 데이터베이스
- 특징: 전 세계에 분산되어 있어 한 곳이 고장 나도 전체 인터넷이 마비되지 않으며, 캐싱을 통해 속도를 최적화함.
