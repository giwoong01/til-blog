PRG(Post-Redirect-Get) 패턴은 웹 개발에서 새로고침으로 인한 폼의 중복 제출을 방지하기 위해 사용하는 가장 기초적이고 중요한 설계 패턴입니다.

## 1. 문제 상황: PRG 패턴이 없을 때

사용자가 상품을 결제하기 위해 폼을 작성하고 제출 버튼을 누른 상황을 가정해 보겠습니다.

1. 사용자가 브라우저에서 POST 요청으로 결제 데이터를 서버에 보냅니다.
2. 서버는 결제를 처리하고, 성공했다는 의미로 “결제 완료” HTML 화면을 즉시 응답(200 OK)해 줍니다.
3. 이때 사용자가 결과 화면에서 무심코 브라우저의 새로고침 버튼을 누릅니다.
4. 브라우저는 가장 마지막에 수행했던 동작인 POST 요청을 서버로 한 번 더 보냅니다. (브라우저에서 “양식을 다시 제출하시겠습니까?” 라는 경고창이 뜨는 것이 바로 이 상황입니다.)
5. 결과적으로 서버에서 결제 로직이 한 번 더 실행되어 중복 결제가 발생합니다.

## 2. 해결책: RPG 패턴의 동작 방식

이 문제를 해결하기 위해, 데이터 저장(POST)과 화면 응답(GET) 사이에 ‘리다이렉트’라는 쿠션을 하나 끼워 넣는 것이 PRG 패턴의 핵심입니다.

- Post (제출): 사용자가 폼 데이터를 POST 방식으로 서버에 전송합니다. 서버는 데이터베이스에 데이터를 저장하거나 결제를 처리합니다.
- Redirect (방향 전환): 서버는 처리가 끝난 후 HTML 화면을 바로 그려서 주는 대신, “작업이 잘 끝났으니, 결과 화면은 저기(다른 URL)로 가서 확인해!” 라며 상태 코드 302 Found (또는 303 See Other)를 응답합니다.
- Get (조회): 브라우저는 서버가 지시한 새로운 URL로 자동으로 GET 요청을 보내 결과 화면을 받아옵니다.

이제 사용자가 결과 화면에서 새로고침을 누르더라도, 브라우저의 마지막 동작은 POST가 아니라 GET 요청이므로 데이터가 중복으로 저장되지 않고 단순히 화면만 다시 불러오게 됩니다.

## 정리

PRG 패턴은 “데이터를 조작하는 위험한 요청(POST)이 끝난 후에는, 반드시 사용자 브라우저를 안전한 조회 페이지(GET)로 강제로 이동(Redirect)시켜서 새로고침 사고를 막는 기법입니다.
