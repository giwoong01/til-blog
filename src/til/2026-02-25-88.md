스프링 트랜잭션 전파 속성은 이미 트랜잭션이 진행 중인 상태에서 새로운 트랜잭션 메소드를 호출할 때, 이 새로운 메소드가 트랜잭션을 어떻게 처리할지 결정하는 설정입니다.

메소드 A(트랜잭션 적용됨) 내부에서 메소드 B(트랜잭션 적용됨)를 호출할 때, B가 A의 트랜잭션에 합류할지, 아니면 자신만의 완전히 독립적인 트랜잭션을 생성할지 등을 제어합니다.

## 1. 전파 속성 7가지 종류

스프링은 @Transactional(propagation = …) 속성을 통해 총 7가지의 전파 옵션을 제공합니다.

1. REQUIRED (기본값)
   - 동작 방식: 진행 중인 부모 트랜잭션이 존재하면 그 트랜잭션에 참여(합류)하고, 진행 중인 트랜잭션이 없으면 새로운 트랜잭션을 생성합니다.
   - 특징: 실무에서 가장 많이 사용되는 기본 설정입니다. 여러 메소드가 하나의 물리적 트랜잭션으로 묶이므로, 어느 한 곳에서라도 예외가 발생하면 전체 작업이 롤백됩니다.

1. REQUIRES_NEW
   - 동작 방식: 부모 트랜잭션의 존재 여부와 상관없이 항상 새로운 트랜잭션을 생성합니다.
   - 특징: 만약 부모 트랜잭션이 실행 중이었다면, 해당 트랜잭션을 잠시 보류하고 새로 생성한 자신의 트랜잭션을 먼저 실행합니다. 두 트랜잭션은 완전히 독립적이므로, 자식 트랜잭션에서 예외가 발생하여 롤백되더라도 부모 트랜잭션에는 영향을 주지 않습니다. (예: 메인 비즈니스 로직과 무관하게 반드시 DB에 기록되어야 하는 감사 로그 저장 기능)

1. SUPPORTS
   - 동작 방식: 부모 트랜잭션이 존재하면 참여하고, 없으면 트랜잭션 없이 실행합니다.
   - 특징: 데이터 변경이 없는 단순 조회용 메소드에 주로 사용됩니다.

1. NOT_SUPPORTED
   - 동작 방식: 트랜잭션을 지원하지 않습니다. 부모 트랜잭션이 존재하면 이를 보류하고, 항상 트랜잭션 없이 실행합니다.

1. MANDATORY
   - 동작 방식: 반드시 부모 트랜잭션이 존재해야만 실행됩니다. 부모 트랜잭션이 없다면 `IllegalTransactionStateException` 예외를 발생시킵니다.
   - 특징: 단독으로 실행되면 안 되고, 반드시 상위 비즈니스 로직의 트랜잭션 흐름 속에서만 호출되어야 하는 메소드에 적용합니다.

1. NEVER
   - 동작 방식: 트랜잭션 컨텍스트 내에서 실행되는 것을 거부합니다. 부모 트랜잭션이 존재하면 예외를 발생시킵니다.

7 NESTED

- 동작 방식: 부모 트랜잭션이 존재하면, 부모 트랜잭션 내부에 중첨 트랜잭션을 생성하여 실행합니다. 부모 트랜잭션이 없다면 REQUIRED와 동일하게 작동합니다.
- 특징: 자식 트랜잭션의 롤백은 부모에게 영향을 주지 않지만, 부모 트랜잭션의 롤백은 자식 트랜잭션에 영향을 줍니다. (DB가 JDBC Savepoint 기능을 지원해야만 사용 가능합니다.)

## 2. 코드 예시

```java
@Service
public class OrderService {

    // 기본값 (REQUIRED): 없으면 만들고, 있으면 참여함
    @Transactional
    public void placeOrder() {
        // 1. 주문 로직 실행

        // 2. 로그 저장 로직 호출
        logService.saveLog();
    }
}

@Service
public class LogService {

    // 주문 로직이 롤백되어도 로그는 무조건 남아야 하므로 완전히 독립적인 트랜잭션으로 분리
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveLog() {
        // 로그 저장 로직
    }
}
```

## 3. 논리적 트랜잭션과 물리적 트랜잭션

전파 속성 REQUIRED 속성을 이해하기 위해 반드시 알아야 하는 내부 구조입니다.

1. 물리적 트랜잭션: 실제 데이터베이스 커넥션을 통해 시작되고 커밋/롤백되는 실제 단위입니다.
2. 논리적 트랜잭션: 스프링 트랜잭션 매니저가 내부적으로 관리하는 트랜잭션 단위입니다.

REQUIRED 설정으로 부모 메소드와 자식 메소드가 묶이게 되면, 이들은 각각의 논리적 트랜잭션을 가지지만 1개의 공유된 물리적 트랜잭션을 사용합니다. 스프링은 “모든 논리적 트랜잭션이 정상적으로 커밋되어야만 물리적 트랜잭션을 커밋한다”는 대원칙을 적용합니다.

## 추가

실무에서 가장 빈번하게 마주치는 트러블슈팅 주제 “REQUIRED 상태에서 내부 트랜잭션에 예외가 발생했을 때, 이를 try-catch로 잡아도 발생하는 UnexpectedRollbackException(Rollback-only 마킹 문제)”의 원인과 해결 방법은?

REQUIRED 전파 속성은 여러 개의 논리적 트랜잭션을 1개의 물리적 트랜잭션으로 묶습니다. 문제는 이 내부 논리적 트랜잭션 중 하나라고 예외를 던지면 발생합니다.

### 1. 문제 상황

예외를 try-catch로 잡았는데 왜 롤백될까?

```java
@Service
public class OrderService {
    @Autowired private PaymentService paymentService;

    @Transactional // 논리적 트랜잭션 A (물리적 트랜잭션 시작)
    public void order() {
        // ... 주문 로직 ...

        try {
            paymentService.pay(); // 내부 메서드 호출
        } catch (RuntimeException e) {
            // 결제 실패 시 예외를 무시하고 주문은 강제로 승인하려는 의도 (잘못된 로직)
            System.out.println("결제 실패를 무시하고 커밋을 시도합니다.");
        }
    }
}

@Service
public class PaymentService {
    @Transactional // 논리적 트랜잭션 B (A의 물리적 트랜잭션에 참여)
    public void pay() {
        // ... 결제 로직 ...
        throw new RuntimeException("결제 모듈 오류 발생");
    }
}
```

개발자의 의도는 “결제에서 에러가 나도 try-catch로 잡았으니, order() 메소드는 정상적으로 커밋되겠지?”입니다. 하지만 실제로는 `UnexpectedRollbackException`이 발생하며 전체 데이터가 롤백됩니다.

### 2. 동작 원리: Rollback-only 마킹

1. 논리적 트랜잭션 B (내부) 실패: pay() 메소드에서 RuntimeException이 발생합니다.
2. 마킹: pay() 메소드를 감싸고 있는 트랜잭션 AOP 프록시는 예외를 감지하고, 이들이 공유하고 있는 물리적 트랜잭션 객체 내부에 rollback-only=true라는 플래그를 마킹합니다. (이때 바로 롤백을 수행하는 것은 아닙니다.)
3. 예외 전달 및 처리: 예외가 호출자인 order() 메소드로 던져지고, try-catch 블록에 의해 예외가 처리됩니다.
4. 논리적 트랜잭션 A (외부) 성공 및 커밋 시도: order() 메소드는 정상적으로 끝났으므로, 스프링 트랜잭션 매니저에게 커밋을 요청합니다.
5. 트랜잭션 매니저의 확인: 물리적 트랜잭션을 최종적으로 커밋하기 전, 매니저는 rollback-only 플래그를 확인합니다.
6. 예외 발생: 매니저는 “어? 누군가 이 트랜잭션은 롤백되어야 한다고 마킹해 놨네? 근데 너는 커밋을 하려고해?” 라고 판단하여 `UnexpectedRollbackException` 을 터뜨리고, 물리적 트랜잭션을 강제로 롤백시켜버립니다.

핵심: 하나의 물리적 트랜잭션을 공유하는 상태에서는, 내부 트랜잭션이 단 한 번이라도 실패하여 마킹을 남기면 외부 트랜잭션에서 예외를 수습하더라도 절대 커밋될 수 없습니다.

### 3. 해결 방법

1. REQUIRES_NEW 사용 (물리적 분리)

내부 로직의 실패가 외부 로직에 영향을 주지 않아야 한다면, 내부 메소드에 새로운 물리적 트랜잭션을 할당해야 합니다.

```java
@Service
public class PaymentService {
    // REQUIRES_NEW: 완전히 독립적인 새로운 물리적 트랜잭션 생성
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void pay() {
        throw new RuntimeException("결제 모듈 오류 발생");
    }
}
```

- 결과: pay()의 트랜잭션은 실패하여 롤백되지만, 이 트랜잭션은 order()의 트랜잭션과 물리적으로 다르기 때문에 order()의 물리적 트랜잭션에는 rollback-only 마킹이 남지 않습니다. 따라서 try-catch 후 order()는 정상 커밋됩니다.

1. 예외가 트랜잭션 경계를 넘지 않도록 수정

내부 메소드 안에서 예외를 처리하여, 트랜잭션 AOP 프록시가 예외를 감지하지 못하도록 막는 방법입니다.

```java
@Service
public class PaymentService {
    @Transactional
    public void pay() {
        try {
            // ... 결제 로직 ...
            throw new RuntimeException("결제 모듈 오류 발생");
        } catch (RuntimeException e) {
            // 내부에서 잡고 정상 리턴 (프록시 입장에서는 성공한 것으로 보임)
        }
    }
}
```

- 결과: pay()의 프록시는 예외를 받지 못했으므로 마킹을 하지 않습니다. 전체 트랜잭션이 정상 커밋됩니다. (다만, 비즈니스 로직상 이렇게 처리하는 것이 맞는지 검토가 필요합니다.)

## 정리

- REQUIRED 상태: 논리적 트랜잭션 중 하나라도 실패하면 물리적 트랜잭션 전체가 Rollback-only 상태가 됨.
- 주의점: 실패한 트랜잭션을 try-catch로 잡아서 무마하려고 하면 `UnexpectedRollbackException` 이 발생함.
- 해결: 독립적인 처리가 필요하다면 REQUIRES_NEW로 물리적 트랜잭션을 아예 분리해야 함.
