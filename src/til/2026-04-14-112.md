행 기반 DB와 열 기반 DB의 가장 큰 차이점은 데이터베이스가 하드 디스크에 데이터를 물리적으로 저장하는 방식입니다.

## 1. 행 기반 DB

우리가 흔히 아는 전통적인 관계형 데이터베이스의 방식입니다. 하나의 레코드(행)를 구성하는 모든 컬럼 데이터가 디스크에 연속적으로 묶여서 저장됩니다.

- 저장 예시: [1번, 기웅, 개발자], [2번, 철수, 디자이너]
- 장점: 특정 사용자의 전체 정보(한 행)를 한 번에 조회하거나, 새로운 데이터를 한 건씩 삽입/수정할 때 매우 빠릅니다. 데이터가 모여있어 디스크의 같은 위치에 한 번만 접근하면 되기 때문입니다.
- 단점: 통계나 분석 쿼리를 실행할 때 치명적입니다. 예를 들어 “모든 사용자의 직업 분포”를 알고 싶어 ‘직업’ 컬럼만 읽으려 해도, 물리적으로 묶여있는 ‘이름’, ‘나이’등 불필요한 데이터까지 메모리에 전부 읽어 들여야 하므로 디스크 I/O 낭비가 극심합니다.
- 주요 용도: 실시간 데이터 처리 및 트랜잭션이 빈번한 OLTP(Online Transaction Processing) 환경 (일반적인 웹/앱 백엔드 서비스)
- 대표 DB: MySQL, PostgreSQL, Oracle

## 2. 열 기반 DB

데이터를 행 단위가 아니라, 같은 속성(컬럼)을 가진 데이터끼리 모아서 디스크에 연속적으로 저장하는 방식입니다.

- 저장 예시: [1번, 2번], [기웅, 철수], [개발자, 디자이너]
- 장점: "전체 사용자의 평균 나이", "총매출 합계"처럼 특정 컬럼만 쏙 골라서 읽어야 하는 집계 연산(SUM, AVG 등) 속도가 압도적으로 빠릅니다. 불필요한 데이터를 아예 읽지 않기 때문입니다. 또한, 같은 데이터 타입이 모여 있으므로 데이터 압축률이 매우 뛰어나 저장 공간을 크게 절약합니다.
- 단점: 레코드 한 건을 삽입하거나 수정하려면, 디스크 여러 군데에 흩어진 컬럼 저장소를 각각 찾아가서 변경해야 하므로 트랜잭션(Insert/Update) 처리가 매우 느립니다.
- 주요 용도: 대규모 데이터 분석 및 통계 작업이 필요한 OLAP(Online Analytical Processing) 환경이나 데이터 웨어하우스.
- 대표 DB: Google BigQuery, Amazon Redshift, Apache HBase

## 정리

| **구분**          | **행 기반 DB (Row-oriented)**      | **열 기반 DB (Column-oriented)**         |
| ----------------- | ---------------------------------- | ---------------------------------------- |
| **물리적 저장**   | 레코드(행) 단위로 연속 저장        | 속성(열) 단위로 연속 저장                |
| **최적화 작업**   | 데이터 삽입, 수정, 삭제 (트랜잭션) | 특정 컬럼의 집계, 통계, 분석 (조회 위주) |
| **I/O 효율**      | 한 건의 전체 데이터를 읽을 때 유리 | 수백만 건 중 일부 컬럼만 읽을 때 유리    |
| **주요 아키텍처** | **OLTP** (실시간 서비스용)         | **OLAP** (데이터 웨어하우스/분석용)      |
