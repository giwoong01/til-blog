JPA의 @OneToOne 연관관계에서 `FetchType.LAZY`를 설정할 때 가장 주의해야 할 점은, 연관관계의 주인(Foreign Key를 가진 쪽)이 아닌 반대편 객체에서는 지연 로딩이 동작하지 않고 강제로 즉시 로딩이 발생한다는 것입니다.

이로 인해 예상치 못한 N + 1 쿼리 문제가 발생하여 시스템 성능이 크게 저하될 수 있습니다.

## 1. 발생 원인: 프록시 객체와 Null 확인

JPA가 지연 로딩을 구현하기 위해서는 실제 엔티티 객체 대신 프록시 객체를 필드에 주입해야 합니다.

그런데 엔티티의 필드 값이 원래 null이어야 하는 상황(즉, 연관된 데이터가 없는 상황)이라면, JPA는 프록시 객체조차 생성해서는 안 되고 해당 필드를 순순한 null 상태로 두어야 합니다.

따라서 JPA는 “이 필드에 프록시 객체를 넣을지, 아니면 null을 넣을지”를 결정하기 위해 연관된 데이터의 존재 여부를 반드시 확인해야만 합니다.

## 2. 연관관계의 주인 (지연 로딩 성공)

테이블 구조상 외래 키를 직접 가지고 있는 쪽을 연관관계의 주인이라고 합니다. (예: Member가 Locker_ID를 가짐)

- 동작 방식: 조회 시 자신의 테이블만 읽어와도 외래 키 컬럼의 값을 확인할 수 있습니다.
- 결과: 외래 키가 null이면 연관 객체도 null로 세팅하고, 값이 있으면 프록시 객체를 생성하여 주입합니다. 지연 로딩이 정상적으로 동작합니다.

## 3. 주인이 아닌 반대편 (지연 로딩 실패)

mappeBy 속성을 사용하는 반대편 객체입니다. 이 테이블에는 외래 키가 없습니다. (예: Locker 테이블에는 Member_ID가 없음)

- 동작 방식: 자신의 테이블만 조회해서는 해당 객체와 연결된 데이터가 존재하는지 알 수 없습니다.
- 결과: 데이터 존재 여부를 확인하기 위해 연결된 테이블을 무조건 데이터베이스에 쿼리하여 뒤져봐야 합니다.
- 문제점: 어차피 쿼리를 날려서 확인해야 하므로, JPA는 프록시를 만드는 의미가 없다고 판단하여 실제 데이터를 모두 가져와 버립니다 (즉시 로딩 동작). 이 경우 Locker 목록을 조회할 때마다 연결된 Member를 찾기 위한 쿼리가 수십, 수백 번 추가로 실행되는 N + 1 문제가 발생합니다.

## 4. 해결 방안

1. 외래 키(FK)의 위치 변경
   - 데이터 접근 패턴을 분석하여, 조회가 더 빈번하게 일어나는 핵심 엔티티 쪽에 외래 키를 두어 연관관계의 주인으로 만듭니다. 가장 권장되는 설계입니다.
2. 단방향 매핑으로 제한
   - 양방향 관계가 굳이 필요하지 않다면, 외래키를 가진 쪽에서만 참조하는 단방향 @OneToOne으로 설계를 변경하여 반대편 탐색 자체를 차단합니다.
3. Fetch Join 활용
   - 연관관계의 주인이 아닌 쪽에서 조회를 해야만 하고 지연 로딩 실패가 불가피하다면, JPQL의 JOIN FETCH나 @EntityGraph를 사용하여 처음부터 두 테이블을 한 번의 쿼리로 묶어서 가져옵니다. (N+1 문제 해결)

## 정리

- @OneToOne 양방향 관계에서 mappedBy가 선언된 쪽은 아무리 FetchType.LAZY를 설정해도 EAGER로 동작합니다.
- 이는 JPA가 프록시 객체를 주입할지 null을 넣을지 판단하기 위해 DB 쿼리가 강제되기 때문입니다.
