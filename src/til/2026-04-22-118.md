ThreadLocal은 멀티 쓰레드 환경에서 “오직 현재 실행 중인 쓰레드만이 접근할 수 있는 전용 보관함”을 제공하는 자바의 클래스입니다.

## 1. 사용하는 이유(장점)

1. 동기화(Lock) 없는 쓰레드 안전성 보장

   변수를 여러 쓰레드가 공유하지 않고 각자의 공간에 따로 복사해서 사용하므로, 앞서 배운 동기화나 락을 걸 필요가 없습니다. 성능 저하 없이 안전하게 데이터를 다룰 수 있습니다.

1. 파라미터 전달(배개변수 릴레이) 지옥 탈출

   만약 사용자의 로그인 ID를 A 메소드에서 D 메소드까지 전달해야 한다고 가정해 봅시다. A(id) → B(id) → C(id) → D(id) 처럼 모든 메소드의 파라미터에 ID를 추가해야 하는 번거로움이 생깁니다. 하지만 ThreadLocal에 ID를 담아두면, 파라미터로 넘기지 않아도 D 메소드에서 언제든지 ThreadLocal를 뒤져서 ID를 꺼내 쓸 수 있습니다.

## 2. 사용 시점

스프링 프레임워크를 비롯한 실무 백엔드 개발에서 매우 폭넓게 사용됩니다.

- 사용자 인증 정보 보관: 스프링 시큐리티가 현재 로그인한 사용자의 정보(SecurityContext)를 저장할 때 ThreadLocal을 사용합니다. 덕분에 코드 어디서든 SecurityContextHolder.getContext()로 내 정보를 꺼내볼 수 있습니다.
- 트랜잭션 관리: 스프링이 데이터베이스 트랜잭션을 시작하고 종료할 때, 동일한 DB 커넥션을 유지하기 위해 커넥션 객체를 ThreadLocal에 보관합니다.
- 웹 요청 단위의 정보 유지: 사용자가 접속해서 나갈 때까지 유지되어야 하는 고유한 TraceID나 로그 정보를 담아둘 때 사용합니다.

## 3. 주의 사항: 쓰레드 풀과 메모리 누수

톰캣 같은 웹 서버는 쓰레드를 생성하고 폐기하는 대신 재사용합니다.

1. 상황: 사용자의 A가 요청을 보내 쓰레드 1번이 할당되었습니다. 쓰레드 1번의 ThreadLocal에 ‘사용자 A의 VIP 등급 정보’가 저장됩니다.
2. 재사용: 사용자 A의 요청이 끝나고 쓰레드 1번이 쓰레드 풀로 반납됩니다. 이때 ThreadLocal을 비우지 않았습니다,
3. 대형 사고: 새로운 일반 사용자 B가 요청을 보냈는데, 하필 쓰레드 1번이 다시 할당되었습니다. 쓰레드 1번은 여전히 ‘사용자 A의 VIP 정보’를 주머니에 가지고 있으므로, 일반 사용자 B가 VIP 권한으로 결제를 해버리는 보안 사고가 발생합니다.

### 해결책

이러한 데이터 유출이나 메모리 누수를 막기 위해, 요청 처리가 끝나는 시점(보통 필터나 인터셉터의 afterCompletion 등)에 반드시 ThreadLocal.remove() 메소드를 호출하여 비워주어야 합니다.

## 정리

- ThreadLocal: 쓰레드마다 독립적으로 할당되는 안전한 개인 데이터 보관함
- 장점: 동시성 문제 해결, 파라미터 전달 생략
- 주의점: 쓰레드 풀을 사용할 경우, 작업 이 끝나면 무조건 remove()로 초기화해야 사고를 막을 수 있음.
