@Component, @Controller, @Service, @Repository 이 어노테이션들은 모두 Spring 프레임워크가 컴포넌트 스캔을 통해 Spring IoC 컨테이너에 빈(Bean)으로 등록하도록 지시하는 스테레오타입 어노테이션입니다.

기능적으로는 네 가지 모두 “이 클래스의 인스턴스를 생성해서 Spring이 관리하게 하라”는 의미를 갖습니다.

핵심적인 차이는 역할과 부가 기능에 있습니다.

이 어노테이션들은 일반적으로 3계층 아키텍처의 각 계층을 명시적으로 구분하기 위해 사용됩니다.

### @Component

가장 일반적이고 포괄적인 Spring 컴포넌트 어노테이션입니다.

- 용도
  - Controller, Service, Repository에 해당하지 않는 범용 컴포넌트(빈)를 등록할 때 사용합니다.
- 예시
  - 유틸리티 클래스(Utils), 설정 헬퍼(Helper), 외부 API 연동 모듈 등 특정 계층으로 분류하기 애매한 클래스들
- 핵심
  - @Controller, @Service, @Repository는 모두 내부적으로 @Component 어노테이션을 포함하고 있습니다. 즉, 이 셋은 @Component의 특별한 케이스입니다.

### @Controller (또는 @RestController)

프레젠테이션 레이어 (웹 계층)

- 용도
  - 사용자의 HTTP 요청을 받고, 그 요청을 처리할 @Service로 전달한 뒤, 처리 결과를 사용자에게 응답(View 또는 데이터)으로 반환하는 역할을 합니다.
- 특징
  - Spring MVC 프레임워크가 이 어노테이션을 보고 웹 컨트롤러로 인식하여 DispatcherServlet과 연결합니다.
  - @RestController: @Controller와 @ResponseBody가 합쳐진 어노테이션입니다.
    JSON/XML 등 REST API의 데이터 본문을 반환하는 데 특화되어 있으며, 현재 Sprint Boot 개발에서 가장 많이 사용됩니다.

### @Service

비즈니스 레이어 (서비스 계층)

- 용도
  - 애플리케이션의 핵심 비즈니스 로직을 구현합니다.
- 특징
  - 컨트롤러(@Controller)와 데이터 접근 객체(@Repository) 사이의 중재자 역할을 합니다.
  - 복잡한 비즈니스 규칙, 트랜잭션 처리(@Transactional) 등이 주로 이 계층에서 이루어집니다.
  - 개발자에게 “이 클래스는 DB 접근 로직이나 웹 요청 처리가 아닌, 순수한 핵심 로직을 담고 있습니다.”라는 의미를 명확하게 전달합니다.

### @Repository

퍼시스턴스 레이어 (데이터 접근 계층)

- 용도
  - 데이터베이스에 접근하여 CRUD(Create, Read, Update Delete) 작업을 수행하는 DAO 클래스에 사용됩니다.
- 부가 기능 (가장 중요한 차이점)
  - @Repository는 @Component나 @Service에는 없는 예외 변환 기능을 제공합니다.
  - JPA, JDBC, Hibernate 등에서 발생하는 데이터베이스 고유의 예외를 Spring 프레이무어크가 일관되게 처리할 수 있는 DataAccessException (런타임 예외)으로 자동 변환하여 서비스 계층에 던져줍니다.
  - 이를 통해 @Service 계층은 특정 DB 기술에 종속되지 않고, 일관된 방식으로 예외를 처리할 수 있게됩니다.

### 정리

모두 빈으로 등록되는 것은 동일하지만

1. 가독성 및 역할 분리: 아키텍처의 각 게층을 명확히 구분하여 코드의 가독성과 유지보수성을 높입니다.
2. 부가 기능 활용: @Repository의 예외 변환 기능처럼 Spring이 제공하는 추가적인 혜택을 누릴 수 있습니다.
3. AOP 대상 지정: @Service 계층 전체에 트랜잭션을 적용하는 등 AOP(관점 지향 프로그래밍)의 포인트컷 대상으로 삼기 용이합니다.

개발자는 @Component만 사용하는 대신, 각 클래스의 역할에 맞는 스테레오타입을 명시적으로 사용해야 합니다.
