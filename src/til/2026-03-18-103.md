Java Collections Framework (JCF)에서 ArrayList, HashMap 같은 동적 자료구조를 생성할 때 초기 용량을 지정하는 가장 큰 이유는 ‘불필요한 크기 확장(Resize) 작업으로 인한 심각한 성능 저하와 메모리 낭비를 막기 위해서’입니다.

## 1. 내부 확장(Resize) 메커니즘의 비밀

JCF의 동적 자료구조들은 크기가 무한대로 늘어나는 것이 아닙니다. 내부적으로 고정된 크기의 배열을 사용하며, 데이터가 꽉 차면 다음과 같은 무거운 과정을 거쳐 몸집을 키웁니다.

1. 새로운 배열 생성: 기존 배열보다 1.5배(ArrayList) 또는 2배(HashMap) 더 큰 새로운 배열을 메모리에 새로 할당합니다.
2. 데이터 복사 (Overhead): 기존 배열에 있던 모든 데이터를 새로 만든 널찍한 배열로 일일이 복사합니다.
3. 기존 배열 폐기: 쓸모없어진 예전 배열은 메모리에 방치되다가 가비지 컬렉터에 의해 청소됩니다.

## 2. 초기 용량을 지정했을 때의 이점

만약 우리가 저장할 데이터의 개수가 대략 10,000개라는 것을 알고 있다면, new ArrayList<>(10000) 처럼 처음부터 공간을 확보해 두는 것이 좋습니다.

1. CPU 연산 낭비 방지 (속도 향상)
   - 지정하지 않았을 때: 기본 용량에서 시작하여 10,000개가 될 때까지 수십 번의 “배열 생성 → 데이터 복사 → 배열 폐기” 과정을 반복해야 합니다. 특히 HashMap의 경우, 크기를 늘릴 때 모든 키(Key)의 해시 위치를 다시 계산하는 재해싱이라는 극악의 CPU연산이 발생합니다.
   - 지정했을 때: 이 모든 복사 및 재배치 연산이 0번으로 줄어들어 데이터 삽입 속도가 비약적으로 빨라집니다.
2. 메모리 단편화 및 GC 부하 감소
   - 배열을 계속 확장하다 보면, 메모리 상에 버려지는 과거의 배열들이 쓰레기처럼 쌓이게 됩니다.
   - 자바의 가비지 컬렉터가 이 쓰레기들을 치우기 위해 시스템을 잠심 멈추고 청소를 해야 하므로, 전체 애플리케이션의 성능이 떨어질 수 있습니다. 처음부터 알맞은 크기를 할당하면 이러한 메모리 낭비와 GC 부하를 원천 차단할 수 있습니다.

## 정리

- 크기를 예측할 수 있을 때: 반드시 초기 용량을 지정. (예: DB에서 100건의 회원 목록을 조회해서 리스트에 담을 때)
- 크기를 예측할 수 없을 때: 억지로 아주 큰 값을 주면 오히려 사용하지 않는 메모리를 낭비하게 되므로, 이때는 기본 생성자를 사용하는 것이 좋습니다.
