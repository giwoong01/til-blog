Keep-Alive는 네트워크 통신에서 한 번 맺은 TCP 연결을 끊지 않고 일정 시간 동안 유지하여, 후속 요청에 재사용하는 기술입니다.

주로 웹 통신을 담당하는 HTTP Keep-Alive(L7)와 전송 계층의 TCP Keepalive(L4) 두 가지 괒넘으로 나뉘며, 일반적으로 웹 개발과 아키텍처에서 성능 최적화를 위해 다루는 것은 전자인 HTTP Keep-Alive입니다.

## 1. TCP 핸드셰이크 오버헤드

HTTP는 기본적으로 요청을 보내고 응답을 반으면 연결을 끊는 비연결성 프로토콜입니다.

- Keep-Alive 미사용 시: 클라이언트가 웹 페이지에 접속하여 HTML, CSS, JS, 이미지 파일 등 10개의 리소스를 요청한다면, 매 요청마다 TCP 연결을 맺기 위한 3-Way Handsake와 연결을 끊기 위한 4-Way Handshake를 10번씩 반복해야 합니다. 이는 심각한 네트워크 지연과 서버 CPU 자원 낭비를 유발합니다.

## 2. 동작 원리

1. 연결 유지: 클라이언트가 요청 헤더에 Connection: keep-alive를 포함하여 보내면, 서버는 응답 후에도 소켓을 즉시 닫지 않고 열어둡니다. (참고로 HTTP/1.1부터는 이 옵션이 기본값으로 적용되어 있습니다.)
2. 연결 재사용: 열려 있는 하나의 TCP 경로를 통해 후속 리소스 요청을 연속적으로 주고받습니다.

## 3. 장단점

- 장점 (성능 향상)
    - 불필요한 핸드셰이크 과정이 생략되어 초기 응답 속도가 비약적으로 향상됩니다.
    - 네트워크 혼잡이 감소하고 패킷 손실률이 줄어듭니다.
- 단점 (서버 자원 점유)
    - 연결을 유지한다는 것은 서버의 쓰레드나 소켓 리소스를 계속 차지하고 있음을 의미합니다.
    - 트래픽이 폭주할 때 클라이언트들이 연결만 맺어두고 실제 요청을 하지 않는 유후 상태가 길어지면, 정작 새로 접속하려는 클라이언트가 자원 부족으로 연결하지 못하는 연결 고갈 문제가 발생할 수 있습니다.

## 4. 핵심 설정 파라미터

단점을 보완하기 위해 웹 서버(Nginx, Apache)나 WAS(Tomcat 등)는 무한정 연결을 유지하지 않고, 다음 두 가지 기준을 설정하여 강제로 연결을 회수합니다.

- Keep-Alive Timeout: 요청이 처리된 후, 다음 요청이 올 때까지 기다리는 최대 대기 시간입니다. (예: 60초) 이 시간 내에 새 요청이 없으면 서버가 연결을 끊습니다.
- Max Requests: 하나의 TCP 연결을 통해 처리할 수 있는 최대 요청 횟수입니다. (예: 100회) 100번의 요청을 처리하면 지정된 시간이 남아있어도 연결을 종료합니다.

## 5. 참고: TCP Keepalive (OS 레벨)

HTTP Keep-Alive가 통신 효율을 위한 것이라면, TCP Keepalive는 연결의 유효성 검증을 위한 기능입니다.

통신 중인 상대방 컴퓨터가 네트워크 단절이나 크래시로 인해 비정상적으로 종료되었는지 확인하기 위해, OS 커널 수준에서 주기적으로 빈 패킷을 보내보고 응답이 없으면 끊어진 것으로 간주하여 자원을 정리합니다.
