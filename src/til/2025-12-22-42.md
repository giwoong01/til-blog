REST는 웹의 장점을 최대한 활용하기 위해 제안된 네트워크 아키텍처 스타일입니다.

쉽게 말해, “HTTP 프로토콜을 의도에 맞게 정확히 사용하며, 웹의 자원을 효율적으로 주고 받는 방법론”입니다.

## 1. REST의 핵심 3요소

REST는 다음 세 가지 요소로 구성됩니다.

- 자원 (Resource) - URI
  - 모든 데이터(자원)은 고유한 ID인 URI를 가집니다.
  - 예: /members/1 (1번 회원이라는 자원)
- 행위 (Verb) - HTTP Method
  - 자원에 대해 수행할 작업을 HTTP 메소드로 표현합니다.
  - 주요 메소드: GET, POST, PUT, DELETE
- 표현 (Representation) - Representation of Resource
  - 자원을 주고받을 때의 형태입니다.
  - 클라이언트와 서버가 데이터를 주고받을 때 JSON, XML 등의 형식으로 자원의 상태를 표현합니다.

## 2. REST가 지향하는 바 (RESTful API 설계)

REST를 잘 지킨 API를 “RESTful하다”고 표현합니다. 가장 중요한 것은 URI는 명사를 사용하고, 행위는 HTTP 메소드로 표현한다는 점입니다.

- 잘못된 행위 (행위를 URI에 포함
  - `GET /getMembers` (조회라는 행위가 URI에 들어감)
  - `POST /deleteMember?id=1` (삭제인데 PSOT를 쓰고, URI에 delete가 들어감)
- 올바른 예 (RESTful)
  - `GET /members` 회원 목록 조회
  - `GET /members/1` 1번 회원 상세 조회
  - `POST /members` 회원 생성 (본문에 데이터 포함)
  - `PUT /members/1` 1번 회원 정보 전체 수정
  - `DELETE /members/1` 1번 회원 삭제

## 3. REST의 6가지 특징 (조건)

진정한 REST 아키텍처가 되기 위해 지켜야 할 제약 조건들입니다.

1. 클라이언트-서버 구조 (Client-Server)
   - 역할이 명확하게 구분됩니다. 클라이언트는 UI/UX에 집중하고, 서버는 데이터 관리에 집중합니다.
2. 무상태성 (Stateless)
   - 가장 중요한 특징입니다. 서버는 클라이언트의 상태(Context, 세션 정보 등)를 기억하지 않습니다.
   - 각 요청은 독립적이며, 필요한 모든 정보를 담아서 서버에 보내야 합니다. (서버 확장성에 유리)
3. 캐시 가능 (Cacheable)
   - HTTP의 기존 웹 표준을 그대로 사용하므로, HTTP의 캐싱 기능(Last-Modified, E-Tag 등)을 적용할 수 있습니다.
4. 계층형 구조 (Layered Systrem)
   - 클라이언트는 서버가 직접 연결된 것인지, 중간에 로드 밸런서나 게이트웨이가 있는지 알 수 없으며 알 필요도 없습니다.
5. 유니폼 인터페이스 (Uniform Interface)
   - URI로 지정한 자원에 대한 조작은 통일되고 한정적인 인터페이스로 수행해야 합니다.
6. 자체 표현 구조 (Self-descriptiveness)
   - API 메시지만 보고도 이것이 무슨 요청인지, 어떻게 처리해야 하는지 이해할 수 있어야 합니다.

## 정리

REST는 “HTTP URL로 자원을 명시하고, HTTP Method(GET, POST, PUT, DELETE)로 해당 자원에 대한 행위(CRUD)를 적용하는 것”입니다.
