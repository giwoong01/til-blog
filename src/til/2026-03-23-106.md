데이터베이스에서 동시성 문제(여러 요청이 같은 데이터를 동시에 수정하려 할 때 발생하는 문제)를 해결하기 위한 대표적인 두 가지 접근 방식인 낙관적 락과 비관적 락이 있습니다.

이 둘은 “충돌이 얼마나 자주 일어날 것이라고 예상하는가?”에 대한 태도 차이에서 출발합니다.

## 1. 낙관적 락

“충돌이 별로 안날거야”라고 낙관적으로 가정하는 방식입니다.

실제로 데이터베이스 수준의 락을 걸지 않고, 애플리케이션 레벨에서 버전(Version) 정보를 이용해 동시성을 제어합니다.

- 동작 원리
  1. 테이블에 version(또는 timestamp) 컬럼을 추가합니다.
  2. 데이터를 읽어올 때 이 version 값도 함께 읽어옵니다. (예: version = 1)
  3. 데이터를 수정하고 DB에 반영(UPDATE)할 때, “내가 처음 읽었던 version이 여전히 그대로인지” 확인합니다.
  4. 맞다면 수정을 반영하고 version을 1 증가시킵니다. (version = 2)
  5. 만약 그사이 다른 트랜잭션이 데이터를 수정해서 DB의 version이 2로 바뀌어 있다면? 업데이트는 실패하고 애플리케이션에 예외가 발생합니다.
- 장점: DB 락을 걸지 않기 때문에 데이터 조회 성능이 매우 빠르고 시스템 자원 소모가 적습니다.
- 단점: 충돌이 발생하면 개발자가 직접 재시도 로직을 짜서 처리해야 합니다. 충돌이 빈번할 경우 계속 롤백하고 재시도하느라 오히려 성능이 크게 떨어질 수 있습니다.

## 2. 비관적 락

“무조건 충돌이 날거야”라고 비관적으로 가정하고, 아예 문을 잠가버리는 방식입니다.

데이터베이스가 제공하는 락 기능(예: SELECT … FOR UPDATE)을 직접 사용하여, 내가 데이터를 읽는 순간부터 수정이 끝날 때까지 다른 사람이 아예 접근하지 못하도록 줄을 세웁니다.

- 동작 원리
  1. 트랜잭션 A가 데이터를 읽을 때 락을 획득합니다.
  2. 트랜잭션 B가 같은 데이터를 읽거나 수정하려고 하면, A가 락을 풀(Commit 또는 Rollback) 때까지 무한정 대기(Blocking) 하게됩니다.
  3. A가 작업을 마치고 락을 반환하면, 비로소 B가 데이터를 읽고 작업을 시작합니다.
- 장점: 데이터의 무결성을 가장 확실하고 엄격하게 보장합니다. 충돌이 났을 때 애플리케이션 단에서 재시도 로직을 복잡하게 구현할 필요가 없습니다.
- 단점: 다른 요청들이 대기해야 하므로 동시 처리 성능이 크게 떨어집니다. 또한, 서로가 서로의 락이 풀리기를 기다리는 교착 상태(Deadlock)에 빠질 위험이 높습니다.
