웹 서버와 WAS의 차이는 “어떤 종류의 콘텐츠를 처리하느냐”에 있습니다.

간단히 요약하면 웹 서버는 정적인 파일을 담당하고, WAS는 동적인 로직을 담당합니다.

## 1. 웹 서버 (Web Server)

웹 서버는 하드웨어적으로는 서버 컴퓨터 그 자체를 의미하기도 하지만, 소프트웨어적으로는 클라이언트의 HTTP 요청을 받아 정적인 콘텐츠를 제공하는 프로그램을 말합니다.

- 주요 역할
  - 정적 리소스 제공: HTML, CSS, JavaScript, 이미지, 파일 등 변하지 않는 데이터를 요청받으면, 하드디스크에서 파일을 찾아 그대로 응답합니다.
  - 리버스 프록시: 동적인 요청이 들어오면, 이를 직접 처리하지 않고 뒤에 있는 WAS에게 전달합니다.
- 대표적인 예: Nginx, Apache HTTP Server, IIS

## 2. WAS (Web Application Server)

WAS는 웹 서버의 기능에 더해, 프로그램 코드를 실행하여 데이터를 가공하고 동적인 콘텐츠를 생성하는 미들웨어입니다. ‘웹 컨테이너(Web Container)’ 또는 ‘서블릿 컨테이너’라고도 불립니다.

- 주요 역할
  - 동적 리소스 제공: “내 장바구니 목록 보여줘” 같은 요청은 미리 만들어둔 파일이 없습니다. WAS는 DB를 조회하고 비즈니스 로직을 실행하여, 사용자 맞춤형 HTML이나 JSON 데이터를 즉석에서 생성합니다.
  - 비즈니스 로직 수행: 트랜잭션 관리, DB 연결 관리(커넥션 풀), 보안 로직 등을 수행합니다.
- 대표적인 예: Tomcat, Jetty, JBoss, WebLogic, WebSphere

## 3. 왜 둘을 함께 사용할까?

사실 Tomcat 같은 WAS는 웹 서버의 기능(정적 파일 제공)도 자체적으로 수행할 수 있습니다. 하지만 실무에서는 대부분 [Client ↔ WebServer(Nginx) ↔ WAS(Tomcat) ↔ DB] 형태로 분리해서 사용합니다.

그 이유는 효율성과 안정성 때문입니다.

1. 부하 분산 (Load Balancing)
   - 앞단의 웹 서버(Nginx)가 여러 대의 WAS에게 요청을 골고루 나눠줄 수 있습니다. 하나의 WAS가 죽어도 서비스는 중단되지 않습니다.
2. 성능 최적화
   - 이미지나 CSS 같은 단순 파일은 빠르고 가벼운 웹 서버가 전담하고, WAS는 무거운 비즈니스 로직에만 집중하게 하여 전체적인 성능을 높입니다.
3. 보안 (Security)
   - 실제 로직과 DB가 연결된 WAS를 외부에 직접 노출하지 않고, 웹 서버를 통해서만 접근하게 하여 보안을 강화합니다. (웹 서버가 DMZ 구간, WAS가 내부망에 위치)

## 정리

- Web Server: 정적인 파일(이미지, HTML) 담당. (Nginx)
- WAS: 동적인 로직(DB 조회, 계산) 담당. (Tomcat)
- 함께 쓰는 이유: 정적인 건 웹 서버가, 동적인 건 WAS가 처리하도록 역할을 나누어 성능과 안정성을 높이기 위함.
