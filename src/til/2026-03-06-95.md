IaC(Infrastructure as Code, 코드로서의 인프라)는 서버, 네트워크, 데이터베이스 로드 밸런서 등 시스템 인프라의 생성과 설정을 수동 프로세스(클라우드 콘솔 화면 클릭, 쉘 명령어 직접 입력 등)가 아닌, 컴퓨터가 읽을 수 있는 코드(텍스트 파일) 형태로 작성하여 자동화하고 배포하는 기술입니다.

소프트웨어 개발에서 소스 코드를 관리하는 것과 동일한 방식을 인프라 인력과 운영 환경에 적용한 DevOps의 핵심 실무 기술입니다.

## 1. 작동 방식 (선언형 vs 명령형)

IaC 도구는 인프라를 코드로 정의할 때 주로 두 가지 접근 방식을 사용합니다. 현대의 IaC는 대부분 선언형을 채택하고 있습니다.

### 1. 선언형 - 주류 방식

- 개념: 인프라의 “최종적으로 원하는 상태”만 코드에 명시합니다.
- 작동: 시스템(IaC 도구)이 인프라의 현재 상태와 코드에 정의된 상태를 비교한 뒤, 부족하거나 다른 부분만 알아서 추가, 수정, 삭제합니다.
- 예시: “AWS EC2 서버 3대가 켜져 있어야 해.”라고 적어두면, 현재 서버가 1대일 경우 2대를 새로 만들고, 이미 3대라면 아무 작업도 하지 않습니다.
- 대표 도구: Terraform, AWS CloudFormation

### 2. 명령형

- 개념: 인프라를 구축하기 위해 실행해야 할 “특정 명령의 순서와 방법”을 코드에 명시합니다.
- 작동: 작성된 스크립트가 순차적으로 실행되며 인프라를 변경합니다.
- 단점: 멱등성을 유지하기 위한 예외 처리(이미 서버가 존재하는지 확인하는 로직 등)를 개발자가 직접 다 짜야 해서 코드가 복잡해집니다.

## 2. 장점 및 도입 목적

IaC를 도입함으로써 얻는 기술적 이점은 다음과 같습니다.

1. 멱등성 보장
   - 선언형 IaC의 가장 중요한 특징입니다. 코드를 1번 실행하든 100번 실행하든 결과적으로 항상 동일한 인프라 상태가 유지됩니다. 이는 중복 배포나 예기치 않은 설정 변경을 막아줍니다.
2. 구성 드리프트 방지
   - 여러 관리자가 수동으로 서버 설정을 건드리다 보면, 개발 환경과 운영 환경이 미세하게 달라지는 현상이 발생합니다. IaC는 코드가 곧 인프라의 명세서가 되므로 모든 환경을 완벽하게 동일하게 유지할 수 있습니다.
3. 버전 관리 시스템 통합
   - 인프라 코드를 Git에 올려 관리하므로, 누가 언제 어떤 인프라 설정을 변경했는지 이력을 추적할 수 있고, 장애 발생 시 이전 커밋으로 롤백하기도 매우 쉽습니다.

## 3. 도구

| 분류              | 역할        | 특징                                                                                                | 대표 도구                              |
| ----------------- | ----------- | --------------------------------------------------------------------------------------------------- | -------------------------------------- |
| 인프라 프로비저닝 | 뼈대 만들기 | 클라우드 환경에서 네트워크(VPC)를 뚫고, 물리적인 서버 자원과 DB 등을 생성하고 연결합니다.           | Terraform, AWS, CloudFormation, Pulumi |
| 구성 관리         | 내부 채우기 | 생성된 서버(OS) 내부에 접속하여 패키지를 설치하고, 환경 변수를 설정하며, 애플리케이션을 배포합니다. | Ansible, Chef, Puppet                  |

## 정리

IaC(Infrastructure as Code)는 마우스 클릭 대신 코드로 서버를 찍어내는 기술입니다. 인프라를 Git으로 관리할 수 있게 만들어 협업을 돕고, 선언적 명세를 통해 수백 대의 서버를 항상 동일하고 안전한 상태(멱등성)로 자동 유지하는 것이 핵심입니다.
