Micrometer(마이크로미터)는 JVM 기반 애플리케이션의 성능 메트릭(Metrics)을 측정하고 수집하기 위한 벤더 중립적인 애플리케이션 메트릭 파사드(Facade)입니다.

자바 진영에서 로깅(Loggin)의 표준 인터페이스로 SLF4J를 사용하는 것과 동일한 원리로, 모니터링 시스템을 위한 SLF4J입니다.

## 1. 역할: 모니터링 시스템의 추상화

애플리케이션의 상태(CPU 사용량, DB 커넥션 풀 상태, HTTP 요청 응답 시간 등)를 모니터링하려면 데이터를 수집해서 외부 시스템(Prometheus, Datadog, New Relic 등)으로 보내야 합니다.

- Micrometer 미사용 시: Prometheus용 API, Datadog용 API 등 사용할 모니터링 툴의 전용 라이브러리에 종속적인 코드를 작성해야 합니다. 모니터링 툴을 변경하면 코드도 모두 수정해야 합니다.
- Micrometer 사용 시: 개발자는 Micrometer의 표준 API만 사용하여 메트릭을 기록합니다. 외부 모니터링 시스템으로 데이터를 전송하는 부분은 Micromter가 각 벤더에 맞는 구현체를 통해 알아서 변환하고 처리합니다.

## 2. 주요 메트릭 타입

Micrometer는 데이터를 측정하기 위해 여러 종류의 Meter 인터페이스를 제공합니다.

1. Counter (카운터)
   - 오직 증가만 하는 누적 값을 측정합니다.
   - 사용 예시: 총 HTTP 요청 횟수, 발생한 에러 건수, 주문 완료 횟수 등
2. Gauge (게이지)
   - 특정 시점의 변동 가능한(오르내리는) 현재 값을 측정합니다.
   - 사용 예시: 현재 활성화된 쓰레드 수, DB 커넥션 풀의 여유 개수, 메모리 사용량 등
3. Timer (타이머)
   - 특정 이벤트가 실행되는 데 걸린 시간과 호출 횟수를 동시에 측정합니다.
   - 사용 예시: API 엔드포인트 응답 시간, 특정 메소드의 실행 시간
4. Distribution Summary (분포 요약)
   - 타이머와 비슷하지만, 시간이 아닌 ‘이벤트의 크기’를 측정하고 분포를 제공합니다.
   - 사용 예시: HTTP 요청 페이로드 크기, 한 번의 트랜잭션에서 조회된 레코드 수

## 3. Spring Boot와 마이크로서비스에서의 활용

마이크로서비스 아키텍처(MSA)에서는 수많은 서비스의 상태를 중앙에서 모니터링해야 하므로, Micromoter와 Prometheus + Grafana 조합을 표준처럼 사용합니다.

1. 의존성 추가: Spring Boot 프로젝트에 `spring-boot-starter-actuator`와 `micrometer-registry-prometheus`를 추가합니다.
2. 자동 구성: Spring Boot Actuator가 JVM 메모리, 톰캣 스레드, HikariCP 커넥션 풀 등의 기본 메트릭을 Micrometer를 통해 자동으로 수집합니다.
3. 엔드포인트 노출: `/actuator/prometheus` 엔드포인트를 통해 수집된 데이터를 Prometheus가 읽어갈 수 있는 포맷으로 변환하여 노출합니다.
4. 커스텀 메트릭: 비즈니스 로직 내부에 `@Timed` 어노테이션을 붙이거나 `MeterRegistry` 객체를 주입받아 애플리케이션 특화 메트릭(예: 특정 상품의 조회수)을 추가로 기록할 수 있습니다.

## 4. 기술적 이점

- 관심사 분리: 비즈니스 로직과 모니터링 시스템 간의 결합도를 낮춰 의존성을 분리합니다.
- 다형성: 코드 수정 없이 application.yml 설정과 의존성 변경만으로 모니터링 백엔드를 교체할 수 있습니다.
- 풍부한 내장 메트릭: 캐시, 데이터베이스, 스케줄링, HTTP 클라이언트 등 다양한 기술 스택에 대한 사전 구성된 메트릭 바인딩을 제공합니다.
