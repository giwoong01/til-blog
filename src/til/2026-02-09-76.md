코드 커버리지는 소프트웨어의 테스트 코드가 소스 코드를 어느 정도 실행(검증)했는지를 나타내는 지표입니다.

쉽게 말해, “우리가 짠 테스트가 전체 코드의 몇 %를 건드려 보았는가?”를 수치로 나타낸 것입니다. 이 수치가 높을수록 버그 발생 가능성이 줄어든다고 볼 수 있습니다.

## 1. 측정 기준

코드 커버리지는 “무엇을 기준으로 측정하느냐”에 따라 여러 종류로 나뉩니다. 가장 많이 쓰이는 3가지는 아래와 같습니다.

1. 구문 커버리지
    - 기준: 코드의 “모든 라인"이 한 번 이상 실행되었는가?
    - 계산: (실행된 라인 수 / 전체 라인 수) * 100
    - 특징: 가장 기본적이고 달성하기 쉽지만, 로직의 분기점(조건문)을 완벽하게 테스트하지 못할 수 있습니다.

1. 분기 커버리지
    - 기준: if, switch 등의 “모든 조건 분기”가 한 번 이상 실행되었는가?
    - 예시: if (A > 0) 문장이 있다면, A가 양수일 때와 양수가 아닐 때 두 경우를 모두 테스트해야 100%가 됩니다.
    - 특징: 구문 커버리지보다 더 강력하며, 실무에서 가장 신뢰하는 지표 중 하나입니다.

1. 조건 커버리지
    - 기준: 조건문 내부의 “각 개별 조건식”이 True/False를 모두 가졌는가?
    - 예시: if (A > 0 && B < 5) 라면, A > 0의 T/F, B < 5의 T/F를 각각 따집니다.

## 2. 예시 (구문 vs 분기)

```java
void check(int x) {
    System.out.println("Start");   // 1번 라인
    if (x > 0) {                   // 2번 라인
        System.out.println("Positive"); // 3번 라인
    }
    System.out.println("End");     // 4번 라인
}
```

테스트 케이스: x = 5 하나만 실행했을 때

1. 구문 커버리지: 100%
    - x가 5이므로 if문 안으로 들어갑니다.
    - 1, 2, 3, 4번 라인이 모두 실행되었습니다.
2. 분기 커버리지: 50%
    - x > 0 인 경우는 실행했지만, x ≤ 0 인 경우는 실행하지 않습니다.
    - 만약 else 로직에 버그가 있다면, 구문 커버리지는 100%여도 버그를 발견할 수 없습니다.

## 3. 목표치

일반적으로 IT업계에서는 라인 커버리지 80% 이상을 권장 목표로 삼는 경우가 많습니다.

하지만 100% 달성에 집착하는 것은 위험합니다.

- 숫자를 채우기 위해 의미 없는 테스트를 짤 수 있습니다.
- 커버리지가 100%라고 해서 논리적 오류가 없다는 뜻은 아닙니다.

## 정리

코드 커버리지는 “테스트가 코드를 얼마나 꼼꼼하게 훑었는지 보여주는 성적표”입니다.

- 구문: 줄 단위 체크
- 분기: 선택지 단위 체크

높은 커버리지는 코드 품질을 위한 필요조건이지, 무결성을 보장하는 충분조건은 아닙니다.
