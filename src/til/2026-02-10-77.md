CI/CD 파이프라인은 애플리케이션의 개발부터 배포까지의 모든 과정을 자동화하여, 코드를 변경할 때마다 빌드, 테스트, 배포 과정을 거쳐 사용자에게 빈번하고 안정적으로 서비스를 제공하는 워크플로우를 의미합니다.

개발자가 코드를 작성한 후 운영 서버에 반영되기까지의 일련의 단계를 파이프가 연결된 것처럼 자동화했다는 의미에서 파이프라인이라고 부릅니다.

## 1. 개념

1. CI (Continuous Integration, 지속적 통합)

개발자들이 작성한 코드를 공유 리포지토리(Git 등)에 자주(하루에도 수십 번) 병합하는 것을 의미합니다.

- 핵심 활동: 코드 커밋 → 자동 빌드 → 자동 유닛 테스트
- 목적: 여러 명의 개발자가 동시에 작업할 때 발생하는 코드 충돌을 방지하고, 버그를 조기에 발견하는 것입니다.

1. CD (Continuous Delivery/Deployment, 지속적 제공/배포)

CI 과정을 통과한 코드를 실제 사용자에게 배포하는 단계입니다.

- 지속적 제공: 프로덕션 배포 전 단계까지 자동으로 배포합니다. 실제 운영 서버 배포는 관리자의 승인이 필요합니다.
- 지속적 배포: 테스트를 모두 통과하면 사람의 개입 없이 자동으로 운영 서버에 배포됩니다.

## 2. CI/CD 파이프라인의 일반적인 단계

기술적으로 파이프라인은 보통 아래  4단계로 구성됩니다.

1. Source (소스 단계)
    - 개발자가 코드를 원격 저장소에 push하면 파이프라인이 트리거되어 시작됩니다.
2. Build (빌드 단계)
    - 코드를 실행 가능한 형태로 만듭니다.
    - Java의 경우 컴파일하고 .jar 파일을 생성하거나, Docker 이미지를 빌드하는 과정이 여기에 해당합니다.
3. Test (테스트 단계)
    - 작성된 코드가 의도대로 동작하는지 검증합니다.
    - 유닛 테스트, 통합 테스트를 수행하며, 실패 시 즉시 파이프라인을 중단하고 개발자에게 알림을 보냅니다.
4. Deploy (배포 단계)
    - 빌드 결과물을 실제 서버에 업로드하고 실행합니다.
    - 이 과정에서 무중단 배포 전략이 사용되기도 합니다.

## 3. 도입 효과

- 배포 속도 향상: 수동으로 하던 빌드/배포 작업을 자동화하여 릴리스 주기를 획기적으로 단축합니다.
- 안정성 확보: 사람의 실수를 방지하고, 항상 동일한 배포 과정을 보장합니다.
- 빠른 피드백: 코드를 올리자마자 테스트 결과를 알 수 있어 버그 수정 비용이 줄어듭니다.

## 4. 대표적인 도구

- Jenkins: 가장 오랫동안 사용된 오픈소스 툴. 커스터마이징이 강력하지만 설정이 복잡할 수 있습니다.
- Github Actions: GitHub 저장소와 완벽하게 통합되어 최근 가장 많이 사용되는 도구입니다.
- GitLab CI: GitLab에 내장되어 있으며, 단일 애플리케이션으로 CI/CD를 모두 관리하기 좋습니다.
- CircleCI / Travis CI: 클라우드 기반의 관리형 서비스입니다.
