JPA에서 패치 조인과 페이징 처리를 함께 사용할 때 가장 주의해야 할 점은 일대다(OneToMany)또는 다대다(ManyToMany) 컬렉션에 대한 패치 조인과 페이징을 동시에 적용하면 안 된다는 것입니다.

## 1.문제: 메모리 페이징 (In-Memory Paging)

일대다 관계의 컬렉션을 패치 조인하는 쿼리에 페이징 옵션을 지정하면, 하이버네이트는 데이터베이스 쿼리 레벨(SQL의 LIMIT, OFFSET)에서 페이징을 수행하지 않습니다.

대신 관련된 모든 데이터를 데이터베이스에서 애플리케이션 메모리로 전부 읽어 들인 후, 메모리 내부에서 페이징 처리를 시도합니다.

이때 하이버네이트는 다음과 같은 경고 로그를 출력합니다.

```
WARN: firstResult/maxResults specified with collection fetch; applying in memory!
```

## 2. 메모리 페이징이 발생하는 원인

이 현상은 관계형 데이터베이스의 조인 특성과 객체 매핑의 차이 때문에 발생합니다.

- 데이터 중복 (카테시안 곱): 1개의 부모 엔티티가 3개의 자식 엔티티를 가지고 있을 때, 데이터베이스에서 일대다 조인을 수행하면 부모 엔티티의 데이터가 자식의 수(3개)만큼 중복되어 조회합니다.
- 페이징의 기준 모호: 이 상태에서 데이터베이스에 LIMIT 옵션을 적용하여 페이징을 하면, 기준이 부모엔티티인지 중복된 행인지 데이터베이스는 알 수 없습니다. 특정 부모 엔티티의 자식 데이터가 중간에 잘린채로 조회되어 데이터 정합성이 심각하게 훼손될 수 있습니다.
- 따라서 하이버네티으는 데이터 정합성을 유지하기 위해 SQL에 LIMIT, OFFSET을 걸지 않고, 일단 모든 조인된 결과를 메로리로 가져온 뒤 애플리케이션 단에서 중복을 제거(부모 엔티티 단위로 병합)하고 페이징을 적용하는 방식을 택합니다.

## 3. 부작용: OOM (Out Of Memory)

테이블에 수만, 수백만 건의 데이터가 존재할 경우, 페이징 처리를 했음에도 불구하고 데이터베이스에 있는 모든 데이터를 백엔드 서버의 힙 메모리에 적재하게 됩니다. 이는 결국 메모리 고갈로 이어져 애플리케이션 서버가 다운되는 장애를 유발합니다.

## 4. 해결 방법

컬렉션 페이징 처리 시 N+1 문제를 방지하면서도 안전하게 페이징을 수행하기 위해 다음 방법들을 사용해야 합니다.

1. 다대일(ManyToOne), 일대일(OneToOne) 관계만 패치 조인
   - 컬렉션이 아닌 단일 엔티티 관계는 조인을 하더라도 데이터 중복이 발생하지 않습니다.
   - 따라서 다대일, 일대일 관계는 기존처럼 패치 조인을 사용하여 조회 쿼리를 최적화해도 데이터베이스 레벨에서 정상적으로 페이징이 이루어집니다.
2. hibernate.default_batch_fetch_size (또는 @BatchSize)사용

   일대다 컬렉션을 페이징해야 할 때 가장 권장되는 실무적인 해결책입니다.
   1. 루트 엔티티 페이징: 패치 조인을 사용하지 않고, 루트 엔티티만 SELECT 하여 페이징 처리(LIMIT, OFFSET)를 수행합니다. (이때 컬렉션은 지연 로딩 상태입니다.)
   2. 배치 사이즈 설정: application.yml 등의 설정 파일에

      spring.jpa.properties.hibernate.default_batch_fetch_size: 1000 (보통 100 ~ 1000 사이)을 설정합니다.

   3. IN 절을 통한 지연 로딩 최적화: 이후 코드에서 지연 로딩된 컬렉션 데이터에 접근할 때, 하이버네이트가 설정된 사이즈만큼 부모 엔티티의 기본키(PK)를 모아 SQL의 IN 절로 묶어서 자식 데이터를 조회합니다.

이 방식을 사용하면 N+1 문제를 IN 절을 통해 1번의 추가 쿼리로 완화하면서, 데이터베이스 레벨의 정확한 페이징 처리를 동시에 달성할 수 있습니다.
