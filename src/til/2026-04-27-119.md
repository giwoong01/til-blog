쓰레드 풀 포화 정책, 또는 거절 정책은 쓰레드 풀이 더 이상 새로운 작업을 수용할 수 없는 한계 상태에 도달했을 때, 새로운 인입된 작업을 어떻게 처리할지 결정하는 시스템의 대응 규칙입니다.

## 1. 포화 상태가 발생하는 조건

쓰레드 풀은 다음과 같은 순서로 자원을 모두 소모했을 때 포화 상태로 판정합니다.

1. Core Pool 가동률 100%: 설정된 모든 핵심 쓰레드가 작업을 수행 중임
2. Work Queue 가득 참: 작업 대기열의 용량이 초과되어 더 이상 작업을 적재할 수 없음
3. Maximum Pool 가동률 100%: 임시로 확장할 수 있는 최대 쓰레드까지 모두 생성되어 작업을 수행 중임

이 3가지 조건이 모두 충족된 상태에서 새로운 작업이 쓰레드 풀에 전달되면 포화 정책이 발동됩니다.

## 2. 자바의 4가지 기본 포화 정책

자바의 ThreadPoolExecutor는 RejectedExecutionHandler 인터페이스를 통해 4가지의 명시적인 포화 정책을 제공합니다. 애플리케이션의 트래픽 특성과 데이터의 중요도에 따라 적절한 정책을 선택해야 합니다.

### 1. AbortPolicy (기본값)

- 동작: 새로운 작업의 수용을 즉시 거절하고, 런타임 예외인 RejectedExecutionException을 발생시킵니다.
- 특징: 시스템이 과부하 상태임을 호출자에게 가장 명확하게 알릴 수 있는 방식입니다. 예외 처리를 통해 개발자가 명시적인 실패 복구 로직(예: 사용자에게 에러 메시지 반환)을 구현할 수 있으므로 가장 널리 사용됩니다.

### 2. CallerRunsPolicy

- 동작: 작업을 쓰레드 풀의 내부 쓰레드에 위임하지 않고, execute() 메소드를 호출한 ‘호출자 쓰레드’가 직접 해당 작업을 실행하도록 만듭니다.
- 특징: 예를 들어 톰캣의 요청 수신 쓰레드가 이 작업을 직접 처리하게 되면, 그 쓰레드가 연산을 마칠 때까지 새로운 외부 요청을 받지 못하게 됩니다. 결과적으로 전체적인 시스템의 작업 유입 속도를 강제로 늦추는 쓰로틀링 효과를 발생시켜 시스템의 연쇄적인 다운을 방지합니다.

### 3. DiscardPolicy

- 동작: 큐에 넣을 수 없는 새로운 작업을 아무런 예외 발생 없이 조용히 폐기합니다.
- 특징: 작업이 누락되더라도 전체 시스템의 정합성에 치명적인 영향이 없는 경우에만 매우 제한적으로 사용해야 합니다.

### 4. DiscardOldestPolicy

- 동작: 현재 대기열에서 가장 오래 대기한 작업을 강제로 폐기하고, 새롭게 인입된 작업을 큐에 삽입하여 실행을 시도합니다.
- 특징: 과거의 데이터보다 최신 데이터의 처리가 시스템 목적상 더 중요한 경우에 적합한 정책입니다.

### 정리

안정적인 시스템 설계를 위해서는 서비스의 성격에 맞춰 포화 정책을 명시적으로 설정하는 것이 중요합니다. 데이터의 손실을 절대 허용할 수 없는 결제 시스템 등에서는 주로 AbortPolicty를 통해 명시적으로 실패를 다루거나, CallerRunsPolicy를 통해 유입량을 조절하는 방식을 채택합니다.
