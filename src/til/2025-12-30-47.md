관계형 데이터베이스(RDBMS)와 비관계형 데이터베이스(NoSQL)의 가장 큰 차이는 “데이터를 저장하는 구조(스키마)”와 “확장 방식”에 있습니다.

## 1. 데이터 저장 구조 (Schema)

### RDBMS

- 정형화된 데이터: 데이터를 행과 열로 이루어진 테이블에 저장합니다.
- 엄격한 스키마: 데이터를 넣기 전에 테이블의 구조(컬럼 명, 데이터 타입)를 미리 정의해야 하며, 이 규칙에 맞지 않는 데이터는 저장할 수 없습니다.
- 관계: 테이블 간의 관계를 통해 데이터를 연결하고, JOIN 연산을 통해 복잡한 데이터를 조회합니다.
- 예: MySQL, PostgreSQL, Oracle

### NoSQL

- 비정형/반정형 데이터: 테이블이 아닌 다양한 형태로 저장합니다. 가장 흔한 Document 방식은 JSON과 유사한 형태를 띱니다.
- 유연한 스키마: 구조가 고정되어 있지 않아, 데이터를 넣을 때 자유롭게ㅔ 필드를 추가하거나 뺄 수 있습니다.
- 예: MongoDB, Redis, Cassandra

## 2. 확장성

## RDBMS

기본적으로  수직 확장(Scale-up)에 최적화 되어 있습니다.

- 데이터 무결성과 트랜잭션(ACID)을 강력하게 보장해야 하므로, 여러 서버에 데이터를 쪼개서 분산시키는 것이 기술적으로 어렵고 복잡합니다.

## NoSQL

태생적으로 수평 확장(Scale-out)을 염두에 두고 설계되었습니다.

- 데이터를 여러 서버에 분산 저장하는 것이 매우 쉽고 자연스럽습니다. 트래픽이 늘어나면 서버만 추가하면 됩니다.

## 3. 트랜잭션과 정합성 (ACID vs BASE)

### RDBMS

ACID 특성을 철저히 지켜 데이터의 정합성을 최우선으로 합니다.(금융 데이터 등에 필수)

### NoSQL

CAP 이론에 따라, 가용성이나 분산 가능성을 위해 정합성을 일부 희생하기도 합니다. 이를 BASE 혹은 최종 일관성이라고 합니다.

## 정리

- RDBMS
    - 데이터 구조가 명확하고 변경될 여지가 적을 때
    - 데이터 일관성이 매우 중요할 때 (예: 은행 시스템, 결제 시스템, 재고 관리)
    - 복잡한 연관 관계 조회가 많을 때
- NoSQL
    - 데이터 구조가 자주 변경되거나 알 수 없을 때
    - 아주 많은 양의 데이터를 고속으로 처리해야 할 때 (예: 로그 수집, IoT 센터 데이터)
    - 데이터의 완벽한 정합성보다 바른 읽기/쓰기 속도가 중요할 때 (SNS 피드, 좋아요 수)
