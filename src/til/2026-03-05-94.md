Java의 String 객체는 불변입니다.

한 번 메모리에 생성된 String 객체의 내부 값(문자열)은 프로그램이 종료될 때까지 절대 변경될 수 없습니다.

`String a = “Hello”; a = a + “ World”;` 와 같이 값을 변경하는 것처럼 보이는 코드는, 기존 객체를 수정하는 것이 아니라 “Hello World”라는 완전히 새로운 객체를 생성하여 변수가 그 새 객체를 가리키도록 참조만 바꾸는 것입니다.

자바 설계자들이 String을 불변으로 만든 기술적인 이유는 크게 4가지가 있습니다.

## 1. 메모리 효율성

자바는 문자열 리터럴을 효율적으로 관리하기 위해 힙(Heap) 영역 내에 String Constant Pool(문자열 상수 풀)이라는 특별한 공간을 둡니다.

- `String str1 = “Java”;` 와 `String str2 = “Java”;` 를 선언하면, 두 변수는 메모리상에서 완전히 동일한 `“Java”` 객체를 참조합니다.
- 만약 String이 가변이었다면: str1의 값을 “C++”로 변경했을 때, 같은 메모리를 바라보고 있던 str2의 값도 원치 않게 “C++”로 변경되는 치명적인 문제가 발생합니다. 불변성이 보장되어야만 여러 변수가 하나의 문자열 객체를 안전하게 공유(캐싱)하여 메모리를 절약할 수 있습니다.

## 2. 보안성

문자열은 자바 애플리케이션에서 데이터베이스 연결 URL, 네트워크 포트, 파일 경로 등 시스템의 핵심적이고 민감한 정보를 전달하는 파라미터로 가장 많이 사용됩니다.

- 만양 String이 가변 객체라면, 권한 확인을 통과한 문자열 변수가 실제로 파일이나 네트워크에 접근하기 직전에 다른 악의적인 쓰레드에 의해 값이 변경될 수 있습니다. 불변 객체는 한 번 검증되면 그 값이 변하지않음을 확신할 수 있으므로 보안상 매우 안전합니다.

## 3. 동시성 문제 해결

불변 객체는 상태가 변하지 않으므로, 여러 쓰레드가 동시에 접근하더라도 데이터가 오염될 위험이 전혀 없습니다.

- 즉, String 객체는 본질적으로 쓰레드 안전합니다. 복잡한 동기화 처리 없이도 멀티 쓰레드 환경에서 안전하게 공유하고 사용할 수 있습니다.

## 4. 성능 최적화 (해시코드 캐싱)

String 객체는 HashMap, HashSet 같은 해시 기반 자료구조의 키로 가장 널리 사용됩니다.

- 해시 기반 자료구조는 성능을 위해 객체의 HashCode()를 빈번하게 호출합니다. String은 값이 변하지 않으므로, 처음 hashcode()가 호출될 때 그 값을 한 번 계산하여 내부에 캐싱(저장)해 둡니다.
- 이후에 해시코드를 요청받으면 계산 없이 저장된 값을 즉시 반환하므로 성능이 비약적으로 향상됩니다. 만약 String이 가변이었다면, 값이 바뀔 때마다 해시코드도 달라져야 하므로 캐싱이 불가능했을 것입니다.

## 정리

내부적으로 String 클래스는 문자열 데이터를 담기 위해 final로 선언된 배열을 사용합니다. 또한 Stirng 클래스 자체도 final 클래스여서 상속을 통해 불변성을 깨뜨리는 것을 원천적으로 차단하고 있습니다.
